## 一、流程概述

每章写完并完成 Settler 状态结算后，系统会触发**状态真值持久化校验**（Truth Persistence Validation）。该校验对比新旧 truth files，检测 Settler 的状态结算是否引入了与正文矛盾的变化。校验失败时自动重试结算（注入 validationFeedback），重试仍失败则标记为 `state-degraded`（状态降级），回退到旧真值文件以保证后续章节不基于错误状态写作。

```
章节写入完成
  │
  ├─ Settler 状态结算 → 产出 updatedState / updatedHooks / updatedLedger
  │
  ├─ validateChapterTruthPersistence
  │   ├─ StateValidatorAgent.validate(正文, 旧truth, 新truth, authorityContext)
  │   │   ├─ computeDiff(oldState, newState) → State Card diff
  │   │   ├─ computeDiff(oldHooks, newHooks) → Hooks Pool diff
  │   │   └─ LLM 判定 PASS / FAIL
  │   │
  │   ├─ 校验通过 (passed=true)
  │   │   └─ 确认写入 status="ready-for-review"
  │   │
  │   ├─ 校验失败 (passed=false)
  │   │   └─ repairChapterState (retrySettlementAfterValidationFailure)
  │   │       ├─ 重新调用 Settler.settleChapterState(allowReapply=true, validationFeedback)
  │   │       ├─ 再次 validate
  │   │       ├─ 重试通过 → 使用新结算结果
  │   │       └─ 重试仍失败 → 状态降级 state-degraded
  │   │
  │   └─ 校验异常 (LLM 不可用)
  │       └─ 直接状态降级 state-degraded
  │
  └─ resyncChapterArtifacts → 同步章节产物
```

## 二、核心概念

| 概念 | 说明 |
|------|------|
| **Truth Files** | 持久化的真值文件：`current_state.md`、`pending_hooks.md`、`particle_ledger.md` |
| **State Validator** | LLM 驱动的连续性校验器，对比新旧 truth files 与正文 |
| **Validation Feedback** | 校验失败时构建的反馈文本，注入 Settler 重试 |
| **State Degraded** | 状态降级，回退到旧 truth files，标记需手动修复 |
| **Authority Context** | 跨真值校验上下文：`story_frame.md`、`book_rules.md`、`chapter_summaries.md` |

## 三、validateChapterTruthPersistence

### 3.1 触发时机

在章节写作流程的 **Settler 结算之后、持久化写入之前** 触发。它是章节持久化的最后一道质量门。

### 3.2 输入参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `writer` | `WriterAgent` | 提供 `settleChapterState` 方法（用于修复重试） |
| `validator` | `StateValidatorAgent` | 提供 `validate` 方法 |
| `book` | `BookConfig` | 书籍配置 |
| `bookDir` | `string` | 书籍目录 |
| `chapterNumber` | `int` | 章节号 |
| `title` | `string` | 章节标题 |
| `content` | `string` | 章节正文 |
| `persistenceOutput` | `WriteChapterOutput` | Settler 产出的结算结果（含 updatedState/updatedHooks） |
| `auditResult` | `AuditResult` | 连续性审查结果 |
| `previousTruth` | `PreviousTruth` | 旧真值：oldState/oldHooks/oldLedger |
| `authorityContext` | `*StateValidationAuthorityContext` | 跨真值上下文（可选） |
| `reducedControlInput` | `*ReducedControlInput` | 精简控制输入（可选，用于修复重试） |
| `language` | `LengthLanguage` | 语言（zh/en） |

### 3.3 校验逻辑

```
1. 调用 validator.validate(content, chapterNumber, oldState, newState, oldHooks, newHooks, language, authorityContext)
2. 如果 LLM 调用异常 → 直接进入状态降级
3. 如果有 warnings → 记录日志
4. 如果 passed=false → 调用 repairChapterState
5. 如果 passed=true → 返回校验通过
```

### 3.4 四项对比维度

StateValidatorAgent 通过对比以下四组文件来检测矛盾：

#### 对比 1: current_state.md（当前状态是否被矛盾修改）

检测 Settler 是否在状态结算中做了与正文矛盾的状态变更：
- **状态变更无正文支撑**：truth file 说角色转移到了森林，但正文只写了意图
- **遗漏状态变更**：正文描述了某事发生，但 truth file 没有捕获
- **时间不可能性**：角色位置跳转无过渡，伤势无时间流逝就痊愈
- **追溯修改**：truth file 变更暗示之前章节发生的事，而非当前章节

#### 对比 2: pending_hooks.md（伏笔是否被错误回收/丢失）

检测伏笔异常：
- **伏笔消失**：hook 被标记为 resolved 但正文没有提及
- **新伏笔无依据**：新出现的 hook 在正文中没有基础
- **伏笔追溯**：truth file 变更暗示之前章节的伏笔活动

#### 对比 3: story_frame.md（世界观铁律是否被违背）

通过 Authority Context 注入 `story_frame.md`（世界观设定），检测：
- **编号规则冲突**：新 truth files 中的编号规则与 story_frame 矛盾
- **命名法则冲突**：命名定律、等级、身份、地点、关系标签与 story_frame 矛盾
- **优先级规则**：当前章节正文 > 运行时 truth files > story_frame/book_rules > 旧 story_bible

#### 对比 4: book_rules.md（规则是否被违反）

通过 Authority Context 注入 `book_rules.md`（书籍规则），检测：
- **规则违背**：新 truth files 违反了 book_rules 中定义的约束
- **规则冲突**：truth file 变更与 book_rules 中编号/命名/身份/地点/关系标签冲突

### 3.5 校验结果分支

```
validate()
  │
  ├─ 异常（LLM 不可用）
  │   └─ 返回 {passed: true, warnings: []} + chapterStatus="state-degraded"
  │       + degradedIssues=[{severity: warning, category: "state-validation", ...}]
  │       + persistenceOutput 回退到旧 truth
  │
  ├─ passed=true
  │   └─ 如果有 warnings → 记录日志
  │       返回 {passed: true, warnings, chapterStatus: null}
  │
  └─ passed=false
      └─ 调用 repairChapterState
          ├─ recovered → 使用新结算结果
          └─ degraded → chapterStatus="state-degraded"
```

## 四、StateValidatorAgent.validate

### 4.1 Diff 计算

Validator 首先计算新旧 truth files 的行级 diff，只将变化部分发给 LLM：

```go
// 伪代码
func computeDiff(oldText, newText, label string) string {
    if oldText == newText {
        return "" // 无变化，跳过
    }
    oldLines := filterNonEmpty(splitLines(oldText))
    newLines := filterNonEmpty(splitLines(newText))

    added := diff(newLines, oldLines)   // newLines 中有但 oldLines 没有的
    removed := diff(oldLines, newLines) // oldLines 中有但 newLines 没有的

    if len(added) == 0 && len(removed) == 0 {
        return ""
    }
    // 构造 diff 文本：
    // ### {label}
    // Removed:
    // - {line}
    // Added:
    // + {line}
    return buildDiffText(label, removed, added)
}
```

**关键逻辑**：如果 stateDiff 和 hooksDiff 都为空（无变化），直接返回 `{passed: true, warnings: []}`，不调用 LLM。

### 4.2 Authority Context

Authority Context 是跨真值校验的权威上下文，包含三部分：

```
## Authority / Cross-Truth Context
Authority priority: current chapter text > runtime truth files/current summaries
> story_frame/book_rules > legacy story_bible intro or marketing-style prose.
If the current chapter establishes a numbered/name mapping, new truth files must
follow that mapping instead of preserving an older intro-only version.

### story_frame / legacy story_bible excerpt
{storyFrame or "(empty)"}

### book_rules excerpt
{bookRules or "(empty)"}

### recent chapter_summaries excerpt
{chapterSummaries or "(empty)"}
```

### 4.3 LLM 判定协议

Validator 使用**极简判决协议**（非 JSON），降低解析失败率：

```
System Prompt 核心要点：
- 检测 6 类矛盾：
  1. 状态变更无正文支撑
  2. 遗漏状态变更
  3. 时间不可能性
  4. 伏笔异常
  5. 追溯修改
  6. 跨真值关键设定冲突

- FAIL 只用于硬矛盾（与正文直接冲突的事实）
- 不对以下情况 FAIL（应为 PASS + warning）：
  - 略超前于正文的推断
  - state card 未捕获的细节遗漏
  - 基于正文的合理外推
  - 不与正文矛盾的伏笔管理差异

输出格式：
- 第一行：PASS 或 FAIL（仅此词）
- 后续行：每行一条 warning，可选 [category] 前缀
- 无问题则仅输出：PASS
```

温度参数：`temperature=0.1`（低温保证判定一致性）。

### 4.4 结果解析

解析器支持两种格式：

**格式 1：极简协议**
```
PASS
[unsupported_change] State card says character moved to the forest, but text only shows intent
[minor] Hook H03 advanced but text mention is brief
```

**格式 2：JSON（兼容）**
```json
{
  "passed": true,
  "warnings": [
    {"category": "unsupported_change", "description": "..."}
  ]
}
```

解析流程：
1. 尝试提取平衡 JSON 对象并解析
2. 如果不是 JSON，按行解析：
   - 第一行必须为 `PASS` 或 `FAIL`
   - 后续行匹配 `[category] description` 或 `- description` 或纯文本

## 五、repairChapterState（状态修复）

### 5.1 修复流程

当 `validateChapterTruthPersistence` 校验失败时，调用 `retrySettlementAfterValidationFailure`：

```
retrySettlementAfterValidationFailure
  │
  ├─ 日志：状态校验失败，正在仅重试结算层
  │
  ├─ 调用 writer.settleChapterState({
  │     book, bookDir, chapterNumber, title, content,
  │     allowReapply: true,                    ← 允许重新结算同一章节
  │     chapterIntent, contextPackage, ruleStack,
  │     validationFeedback: buildStateValidationFeedback(warnings, language)
  │   })
  │
  ├─ 再次调用 validator.validate(content, chapterNumber,
  │     oldState, retryOutput.updatedState,
  │     oldHooks, retryOutput.updatedHooks, language)
  │
  ├─ 重试校验通过 (passed=true)
  │   └─ 返回 {kind: "recovered", output: retryOutput, validation: retryValidation}
  │
  └─ 重试校验仍失败 (passed=false)
      └─ 返回 {kind: "degraded", issues: buildStateDegradedIssues(warnings, language)}
```

**关键点**：
- `allowReapply=true` 允许 Settler 重新结算同一章节（正常流程禁止重复结算）
- `validationFeedback` 将校验失败的具体矛盾注入 Settler prompt，指导其修正

### 5.2 validationFeedback 构建

```go
func buildStateValidationFeedback(warnings []ValidationWarning, language LengthLanguage) string {
    if len(warnings) == 0 {
        if language == LangEN {
            return "The previous settlement contradicted the chapter text. " +
                "Reconcile truth files strictly to the body."
        }
        return "上一次状态结算与正文矛盾。请严格以正文为准修正 truth files。"
    }

    if language == LangEN {
        lines := []string{
            "The previous settlement failed validation. " +
                "Fix these contradictions against the chapter body:",
        }
        for _, w := range warnings {
            lines = append(lines, fmt.Sprintf("- [%s] %s", w.Category, w.Description))
        }
        return strings.Join(lines, "\n")
    }

    lines := []string{"上一次状态结算未通过校验。请对照正文修正以下矛盾："}
    for _, w := range warnings {
        lines = append(lines, fmt.Sprintf("- [%s] %s", w.Category, w.Description))
    }
    return strings.Join(lines, "\n")
}
```

### 5.3 修复结果判定

| 重试结果 | 说明 | 后续动作 |
|----------|------|----------|
| `recovered` | 重试结算后校验通过 | 使用新的 `retryOutput` 作为持久化输出 |
| `degraded` | 重试结算后校验仍失败 | 标记 `state-degraded`，回退旧 truth |

## 六、状态降级机制

### 6.1 state-degraded 标记

当校验失败且修复重试也失败时，章节状态被标记为 `state-degraded`：

- `chapterStatus = "state-degraded"`
- `degradedIssues` = 修复失败产生的问题列表
- 持久化输出回退到旧 truth files
- 审查结果注入降级问题

### 6.2 降级持久化输出

```go
func buildStateDegradedPersistenceOutput(
    output WriteChapterOutput,
    oldState, oldHooks, oldLedger string,
) WriteChapterOutput {
    return WriteChapterOutput{
        // 保留章节正文和元数据
        ChapterContent:   output.ChapterContent,
        ChapterTitle:     output.ChapterTitle,
        // 回退所有 truth files 到旧值
        UpdatedState:     oldState,
        UpdatedLedger:    oldLedger,
        UpdatedHooks:     oldHooks,
        // 清空运行时状态增量与快照
        RuntimeStateDelta:    nil,
        RuntimeStateSnapshot: nil,
        // 清空章节摘要更新
        UpdatedChapterSummaries: nil,
    }
}
```

**关键语义**：降级时正文已写入磁盘（不可撤销），但 truth files 回退到旧值，确保后续章节不基于错误状态写作。降级章节需要人工修复后再继续。

### 6.3 降级审查注记

降级章节的 `reviewNote` 字段存储 JSON 格式的降级元数据：

```json
{
  "kind": "state-degraded",
  "baseStatus": "ready-for-review",
  "injectedIssues": [
    "[warning] State says character is dead but chapter text shows them speaking",
    "[warning] New location not mentioned anywhere in chapter text"
  ]
}
```

- `baseStatus`：降级前的原始状态（`ready-for-review` 或 `audit-failed`）
- `injectedIssues`：注入的降级问题列表

解析函数 `parseStateDegradedReviewNote` 用于在后续流程中识别降级章节并恢复其原始状态判断。

## 七、resyncChapterArtifacts

`resyncChapterArtifacts` 在校验/修复完成后重新同步章节产物，确保磁盘上的 truth files 与最终持久化输出一致。

```
resyncChapterArtifacts(bookDir, chapterNumber, persistenceOutput)
  │
  ├─ 写入 current_state.md ← persistenceOutput.UpdatedState
  ├─ 写入 pending_hooks.md ← persistenceOutput.UpdatedHooks
  ├─ 写入 particle_ledger.md ← persistenceOutput.UpdatedLedger
  │
  ├─ 如果 persistenceOutput.RuntimeStateDelta != nil:
  │   ├─ buildRuntimeStateArtifacts(bookDir, delta, language, allowReapply)
  │   │   ├─ loadRuntimeStateSnapshot(bookDir)
  │   │   ├─ arbitrateRuntimeStateDeltaHooks(hooks, delta)
  │   │   ├─ applyRuntimeStateDelta(snapshot, resolvedDelta)
  │   │   └─ 渲染 Markdown 投影
  │   ├─ saveRuntimeStateSnapshot(bookDir, next)
  │   └─ 写入 state/ 下的 JSON 文件
  │
  ├─ 如果 persistenceOutput.UpdatedChapterSummaries != nil:
  │   └─ 写入 chapter_summaries.md
  │
  └─ 写入 chapters/{N}_{title}.md ← persistenceOutput.ChapterContent
```

**降级场景**：如果 `persistenceOutput.RuntimeStateDelta` 为 nil（降级输出），跳过结构化状态更新，仅同步 Markdown truth files 到旧值。

## 八、validateRuntimeState（结构化校验）

除了 LLM 驱动的语义校验外，还有一个纯规则的结构化校验器 `validateRuntimeState`，用于校验 JSON 格式的运行时状态文件：

```
validateRuntimeState(manifest, currentState, hooks, chapterSummaries)
  │
  ├─ 校验 manifest.json 符合 StateManifestSchema
  ├─ 校验 current_state.json 符合 CurrentStateStateSchema
  ├─ 校验 hooks.json 符合 HooksStateSchema
  ├─ 校验 chapter_summaries.json 符合 ChapterSummariesStateSchema
  │
  ├─ hooks 去重检查：hookId 不能重复
  ├─ chapterSummaries 去重检查：chapter 不能重复
  ├─ 一致性检查：currentState.chapter <= manifest.lastAppliedChapter
  │
  └─ 返回 issues 列表（code/message/path）
```

该校验在 `loadRuntimeStateSnapshot` 和 `applyRuntimeStateDelta` 中自动执行，发现问题立即抛错。

## 九、完整 Go 接口定义

```go
package statevalidation

import (
	"context"
)

// ============================================================================
// 核心数据结构
// ============================================================================

// ValidationWarning 校验警告
type ValidationWarning struct {
	Category   string `json:"category"`
	Description string `json:"description"`
}

// ValidationResult 校验结果
type ValidationResult struct {
	Warnings []ValidationWarning `json:"warnings"`
	Passed   bool                `json:"passed"`
}

// StateValidationAuthorityContext 跨真值校验上下文
type StateValidationAuthorityContext struct {
	StoryFrame      *string `json:"storyFrame,omitempty"`
	BookRules       *string `json:"bookRules,omitempty"`
	ChapterSummaries *string `json:"chapterSummaries,omitempty"`
}

// PreviousTruth 旧真值文件
type PreviousTruth struct {
	OldState  string `json:"oldState"`
	OldHooks  string `json:"oldHooks"`
	OldLedger string `json:"oldLedger"`
}

// ReducedControlInput 精简控制输入（用于修复重试）
type ReducedControlInput struct {
	ChapterIntent  string         `json:"chapterIntent"`
	ContextPackage ContextPackage `json:"contextPackage"`
	RuleStack      RuleStack      `json:"ruleStack"`
}

// WriteChapterOutput Settler 产出的结算结果
type WriteChapterOutput struct {
	ChapterContent           string                 `json:"chapterContent"`
	ChapterTitle             string                 `json:"chapterTitle"`
	UpdatedState             string                 `json:"updatedState"`
	UpdatedLedger            string                 `json:"updatedLedger"`
	UpdatedHooks             string                 `json:"updatedHooks"`
	UpdatedChapterSummaries  *string                `json:"updatedChapterSummaries,omitempty"`
	RuntimeStateDelta        *RuntimeStateDelta     `json:"runtimeStateDelta,omitempty"`
	RuntimeStateSnapshot     *RuntimeStateSnapshot  `json:"runtimeStateSnapshot,omitempty"`
}

// AuditIssue 审查问题
type AuditIssue struct {
	Severity   string `json:"severity"`   // critical / warning / info
	Category   string `json:"category"`
	Description string `json:"description"`
	Suggestion string `json:"suggestion"`
}

// AuditResult 审查结果
type AuditResult struct {
	Issues []AuditIssue `json:"issues"`
}

// LengthLanguage 语言类型
type LengthLanguage string

const (
	LangZH LengthLanguage = "zh"
	LangEN LengthLanguage = "en"
)

// ============================================================================
// StateValidatorAgent
// ============================================================================

// StateValidatorAgent 状态校验器 Agent
type StateValidatorAgent interface {
	// Name 返回 Agent 名称
	Name() string

	// Validate 对比新旧 truth files，检测与正文的矛盾
	Validate(
		ctx context.Context,
		chapterContent string,
		chapterNumber int,
		oldState string,
		newState string,
		oldHooks string,
		newHooks string,
		language LengthLanguage,
		authorityContext *StateValidationAuthorityContext,
	) (*ValidationResult, error)
}

// StateValidatorConfig 校验器配置
type StateValidatorConfig struct {
	// LLM 客户端
	LLMClient LLMClient
	// 日志记录器
	Logger Logger
}

// ============================================================================
// validateChapterTruthPersistence 入口
// ============================================================================

// ValidateChapterTruthPersistenceParams 校验参数
```
