## 一、流程概述

回炉重写是写作流水线的修复阶段，在章节审查循环（Phase 5）内部被调用。当 ContinuityAuditor 审计发现问题且未通过时，审查循环会创建 ReviserAgent 实例，传入审计 issues，让 ReviserAgent 根据问题类型选择合适的修订策略。

ReviserAgent 支持 6 种修订模式，覆盖从"只改一个词"到"整章重写"的全谱系：

```
AuditResult.issues
  │
  ▼
┌─────────────────────────────────────────────┐
│ ReviserAgent.reviseChapter(mode)             │
│                                              │
│  mode 选择：                                  │
│  ┌──────────┐  审查循环默认调用               │
│  │ auto     │←─ 根据 issues 自动路由          │
│  └────┬─────┘    patch-only / rewrite-only   │
│       │          / allow-full                │
│       ▼                                      │
│  ┌──────────┬──────────┬──────────┐          │
│  │ patch    │ rewrite  │  full    │          │
│  │ (局部)   │ (全章)   │  (二选一) │          │
│  └──────────┴──────────┴──────────┘          │
│                                              │
│  手动调用模式（CLI/API）：                     │
│  ┌──────────┬──────────┬──────────┐          │
│  │ polish   │ rewrite  │ rework   │          │
│  │ (润色)   │ (改写)   │ (重写)   │          │
│  ├──────────┼──────────┴──────────┤          │
│  │anti-detect│   spot-fix         │          │
│  │(反检测)  │   (定点修复)         │          │
│  └──────────┴─────────────────────┘          │
└─────────────────┬───────────────────────────┘
                  ▼
┌─────────────────────────────────────────────┐
│ 加载上下文（10 个 truth files + 题材规则）    │
│ → 构建 system prompt（按 mode 不同）         │
│ → 构建 user prompt（含 issues + 正文）       │
│ → LLM 生成（temperature=0.3）                │
│ → parseOutput 解析                           │
└─────────────────┬───────────────────────────┘
                  ▼
          ReviseOutput
  { revisedContent, fixedIssues, updatedState, ... }
```

**核心设计原则**：
1. **按问题类型路由**：auto 模式根据 issues 的 `repairScope` 和 `category` 自动选择 patch 还是 rewrite
2. **最小改动原则**：局部问题用 PATCHES，不整章重写；结构问题才用 REVISED_CONTENT
3. **字数独立处理**：auto 模式不处理字数（由 normalizeIfHardDrift 专步处理），legacy 模式才有字数护栏
4. **governed 模式合并**：governed 模式下，修订后的伏笔池按 key 合并到原表，不覆盖无关行

## 三、6 种修订模式详解

### 3.1 auto：自动模式

**审查循环默认调用模式**。根据 issues 的 `repairScope` 和 `category` 自动选择策略。

```go
// MODE_DESCRIPTIONS["auto"] = "" （auto 模式使用独立的 buildAutoSystemPrompt）
```

**auto 模式的三种输出路径**：

| AutoOutputMode | 触发条件 | 输出格式 | 说明 |
|----------------|----------|----------|------|
| `patch-only` | 所有阻塞问题都是 local 级 | 仅 PATCHES | 局部补丁，不改整章 |
| `rewrite-only` | 有任一 structural 级问题 | 仅 REVISED_CONTENT | 全章重写 |
| `allow-full` | 混合/未知问题集 | PATCHES 或 REVISED_CONTENT | 让 LLM 自行选择 |

**auto 模式 system prompt 特点**：
- 不使用 legacy 的 `MODE_DESCRIPTIONS`
- 使用独立的 `buildAutoSystemPrompt`，包含：
  - PATCHES 和 REVISED_CONTENT 的选择指引
  - 路由指令（routing directive）：根据 autoOutputMode 强制输出路径
  - 7 条修稿原则（修根因/伏笔同步/不改走向/保持风格/情绪外化/对话区分/升级叠加）
  - 小目标周期修稿指引（后效阶段/爆发阶段/日常段落的处理方式）

**字数约束**（auto 模式特有）：
- 当 lengthSpec 存在时，REVISED_CONTENT 必须在 `[softMin, softMax]` 区间内
- 这是硬性约束，不可突破
- 但 auto 模式不使用 legacy 的字数护栏（`lengthGuardrail`），因为字数由 normalize 独立处理

### 3.2 polish：润色模式

**只改表达，不改事实**。

```markdown
润色：只改表达、节奏、段落呼吸，不改事实与剧情结论。
禁止：增删段落、改变人名/地名/物品名、增加新情节或新对话、改变因果关系。
只允许：替换用词、调整句序、修改标点节奏
```

**适用场景**：
- 文风统一性调整
- 节奏微调（句式长短交替）
- 标点规范化（如去除破折号）
- 段落呼吸感优化

**输出格式**：REVISED_CONTENT（完整正文）

### 3.3 rewrite：改写模式

**重组问题段落，优先保留原文**。

```markdown
改写：允许重组问题段落、调整画面和叙述力度，但优先保留原文的绝大部分句段。
除非问题跨越整章，否则禁止整章推倒重写；只能围绕问题段落及其直接上下文改写，
同时保留核心事实与人物动机
```

**适用场景**：
- 某些段落需要重新组织（如视角混乱、叙事顺序不佳）
- 画面感不足需要增强描写
- 叙述力度需要调整（如高潮段落不够紧凑）

**与 rework 的区别**：rewrite 优先保留原文，只改问题段落；rework 可以重构场景推进方式。

**输出格式**：REVISED_CONTENT（完整正文）

### 3.4 rework：重写模式

**可重构场景推进和冲突组织**。

```markdown
重写：可重构场景推进和冲突组织，但不改主设定和大事件结果
```

**适用场景**：
- 整章节奏需要重新组织
- 场景推进方式需要调整（如改为倒叙、插入闪回）
- 冲突组织需要重构（如调整对抗顺序）
- 但**不能改变**：主设定（世界观/力量体系）、大事件结果（谁赢谁输/谁死谁活）

**与 rewrite 的区别**：rework 允许重构场景推进方式，rewrite 只在现有场景框架内调整。

**输出格式**：REVISED_CONTENT（完整正文）

### 3.5 anti-detect：反检测模式

**降低 AI 生成可检测性**。

```markdown
反检测改写：在保持剧情不变的前提下，降低AI生成可检测性。

改写手法（附正例）：
1. 打破句式规律：连续短句 → 长短交替，句式不可预测
2. 口语化替代：✗"然而事情并没有那么简单" → ✓"哪有那么便宜的事"
3. 减少"了"字密度：✗"他走了过去，拿了杯子" → ✓"他走过去，端起杯子"
4. 转折词降频：✗"虽然…但是…" → ✓ 用角色内心吐槽或直接动作切换
5. 情绪外化：✗"他感到愤怒" → ✓"他捏碎了茶杯，滚烫的茶水流过指缝"
6. 删掉叙述者结论：✗"这一刻他终于明白了力量" → ✓ 只写行动，让读者自己感受
7. 群像反应具体化：✗"全场震惊" → ✓"老陈的烟掉在裤子上，烫得他跳起来"
8. 段落长度差异化：不再等长段落，有的段只有一句话，有的段七八行
9. 消灭"不禁""仿佛""宛如"等AI标记词：换成具体感官描写
```

**适用场景**：
- 章节被 AI 检测工具标记为"AI生成"
- 需要降低 AI 写作痕迹
- 保持剧情完全不变，只改表达方式

**9 种改写手法**：
1. 打破句式规律（连续短句→长短交替）
2. 口语化替代（书面转折词→角色内心吐槽）
3. 减少"了"字密度
4. 转折词降频（虽然但是→直接动作）
5. 情绪外化（标签词→身体动作）
6. 删叙述者结论（让读者自己感受）
7. 群像反应具体化（全场→个人）
8. 段落长度差异化（不再等长）
9. 消灭 AI 标记词（仿佛/不禁/宛如→感官描写）

**输出格式**：REVISED_CONTENT（完整正文）

### 3.6 spot-fix：定点修复模式

**只改审稿意见指出的具体句子**。

```markdown
定点修复：只修改审稿意见指出的具体句子或段落，其余所有内容必须原封不动保留。
修改范围限定在问题句子及其前后各一句。禁止改动无关段落
```

**适用场景**：
- 只有零星几处需要修改
- 修改范围非常精确（一句话或一个段落）
- 需要保持正文其余部分完全不变

**输出格式**：PATCHES（局部补丁，不输出整章）

```markdown
=== FIXED_ISSUES ===
(逐条说明修正了什么，一行一条；如果无法安全定点修复，也在这里说明)

=== PATCHES ===
--- PATCH 1 ---
TARGET_TEXT:
(必须从原文中精确复制、且能唯一命中的原句或原段)
REPLACEMENT_TEXT:
(替换后的局部文本)
--- END PATCH ---

=== UPDATED_STATE ===
(更新后的完整状态卡)

=== UPDATED_HOOKS ===
(更新后的完整伏笔池)
```

**关键约束**：
- TARGET_TEXT 必须能在原文中**唯一命中**
- 如果需要大面积改写，说明无法安全 spot-fix，让 PATCHES 留空
- 禁止输出整章改写

## 四、auto 模式路由算法

auto 模式的核心是 `resolveAutoOutputMode` 函数，根据 issues 的 `repairScope` 和 `category` 决定输出路径。

### 4.1 路由决策流程

```go
// resolveAutoOutputMode 决定 auto 模式的输出路径
func resolveAutoOutputMode(issues []AuditIssue) AutoOutputMode {
    if len(issues) == 0 {
        return AutoOutputAllowFull
    }

    // 1. 优先看 repairScope（LLM 标注的路由提示）
    scopedBlocking := filter(issues, func(i AuditIssue) bool {
        return i.Severity != "info" && i.RepairScope != ""
    })
    if len(scopedBlocking) > 0 {
        // 有 structural 级问题 → rewrite-only
        if any(scopedBlocking, func(i AuditIssue) bool { return i.RepairScope == "structural" }) {
            return AutoOutputRewriteOnly
        }
        // 全部是 local 级 → patch-only
        blockingCount := count(issues, func(i AuditIssue) bool { return i.Severity != "info" })
        if len(scopedBlocking) == blockingCount && all(scopedBlocking, func(i AuditIssue) bool { return i.RepairScope == "local" }) {
            return AutoOutputPatchOnly
        }
    }

    // 2. 回退到 category 正则匹配
    blocking := filter(issues, func(i AuditIssue) bool { return i.Severity != "info" })
    if len(blocking) == 0 {
        return AutoOutputPatchOnly // 只有 info 级 → 至多局部润色
    }

    structuralCount := count(blocking, isStructural)
    localOnlyCount := count(blocking, isLocal)

    // 有结构问题 → rewrite-only
    if structuralCount > 0 {
        return AutoOutputRewriteOnly
    }
    // 全部是局部问题 → patch-only
    if localOnlyCount == len(blocking) {
        return AutoOutputPatchOnly
    }
    // 混合/未知 → allow-full
    return AutoOutputAllowFull
}
```

### 4.4 路由指令（routing directive）

system prompt 中根据 autoOutputMode 注入不同的路由指令：

| AutoOutputMode | 中文路由指令 |
|----------------|-------------|
| `rewrite-only` | 分流指令：reviewer 报告的阻塞问题属于结构/语义错（人设崩、主线偏、爽点缺、时间线错、伏笔未收、memo 偏离等）。你必须输出 REVISED_CONTENT——禁止输出 PATCHES，这类问题不能靠补丁修复。如果无法安全重写，在 FIXED_ISSUES 里说明并留空 REVISED_CONTENT。 |
| `patch-only` | 分流指令：reviewer 报告的阻塞问题属于局部错（措辞、段落形状、疲劳词、信息越界、知识污染）。你必须只输出 PATCHES——不要整章改写。如果做不出补丁，留空 PATCHES。 |
| `allow-full` | （无路由指令，让 LLM 自行选择） |

### 5.4 输出解析逻辑

```go
// parseOutput 解析修订输出
func (a *ReviserAgent) parseOutput(content string, gp *GenreProfile, mode ReviseMode, originalChapter string, autoOutputMode AutoOutputMode) *ReviseOutput {
    extract := func(tag string) string {
        re := regexp.MustCompile(fmt.Sprintf(`=== %s ===\s*([\s\S]*?)(?==== [A-Z_]+ ===|$)`, tag))
        match := re.FindStringSubmatch(content)
        if len(match) > 1 {
            return strings.TrimSpace(match[1])
        }
        return ""
    }

    fixedRaw := extract("FIXED_ISSUES")
    fixedIssues := splitNonEmptyLines(fixedRaw)

    makeResult := func(revisedContent string, applied bool) *ReviseOutput {
        return &ReviseOutput{
            RevisedContent: revisedContent,
            WordCount:      len([]rune(revisedContent)),
            FixedIssues:    ternary(applied, fixedIssues, []string{}),
            UpdatedState:   extract("UPDATED_STATE") | "(状态卡未更新)",
            UpdatedLedger:  ternary(gp.NumericalSystem, extract("UPDATED_LEDGER") | "(账本未更新)", ""),
            UpdatedHooks:   extract("UPDATED_HOOKS") | "(伏笔池未更新)",
        }
    }

    // auto 模式：按 autoOutputMode 路由
    if mode == ReviseModeAuto {
        switch autoOutputMode {
        case AutoOutputPatchOnly:
            // 仅接受 PATCHES
            patchesRaw := extract("PATCHES")
            patches := parseSpotFixPatches(patchesRaw)
            if len(patches) == 0 {
                return makeResult(originalChapter, false) // 无补丁 → 原文不变
            }
            patchResult := applySpotFixPatches(originalChapter, patches)
            // 应用率 >= 50% 才算成功
            if patchResult.Applied && float64(patchResult.AppliedPatchCount)/float64(len(patches)) >= 0.5 {
                return makeResult(patchResult.RevisedContent, true)
            }
            return makeResult(originalChapter, false)

        case AutoOutputRewriteOnly:
            // 仅接受 REVISED_CONTENT
            revisedContent := extract("REVISED_CONTENT")
            if revisedContent == "" {
                // 结构问题无法重写 → 不回退到 patches，返回原文
                return makeResult(originalChapter, false)
            }
            return makeResult(revisedContent, true)

        case AutoOutputAllowFull:
            // 优先 REVISED_CONTENT，其次 PATCHES
            revisedContent := extract("REVISED_CONTENT")
            if revisedContent != "" {
                return makeResult(revisedContent, true)
            }
            patchesRaw := extract("PATCHES")
            patches := parseSpotFixPatches(patchesRaw)
            if len(patches) > 0 {
                patchResult := applySpotFixPatches(originalChapter, patches)
                if patchResult.Applied && float64(patchResult.AppliedPatchCount)/float64(len(patches)) >= 0.5 {
                    return makeResult(patchResult.RevisedContent, true)
                }
            }
            return makeResult(originalChapter, false) // 两者都空 → 原文不变
        }
    }

    // spot-fix 模式：仅 PATCHES
    if mode == ReviseModeSpotFix {
        patches := parseSpotFixPatches(extract("PATCHES"))
        patchResult := applySpotFixPatches(originalChapter, patches)
        return makeResult(patchResult.RevisedContent, patchResult.Applied)
    }

    // legacy 模式（polish/rewrite/rework/anti-detect）：完整正文
    revisedContent := extract("REVISED_CONTENT")
    if revisedContent == "" {
        return makeResult(originalChapter, false)
    }
    return makeResult(revisedContent, true)
}
```

### 5.5 SpotFix Patch 格式

```go
// SpotFixPatch 定点修复补丁
type SpotFixPatch struct {
    TargetText      string // 原文中要被替换的文本（必须唯一命中）
    ReplacementText string // 替换后的文本
}

// parseSpotFixPatches 解析补丁列表
func parseSpotFixPatches(raw string) []SpotFixPatch {
    patches := []SpotFixPatch{}
    // 按 --- PATCH N --- / --- END PATCH --- 分割
    blocks := splitByPatchDelimiter(raw)
    for _, block := range blocks {
        target := extractSection(block, "TARGET_TEXT:")
        replacement := extractSection(block, "REPLACEMENT_TEXT:")
        if target != "" && replacement != "" {
            patches = append(patches, SpotFixPatch{
                TargetText:      target,
                ReplacementText: replacement,
            })
        }
    }
    return patches
}

// applySpotFixPatches 应用补丁到原文
func applySpotFixPatches(original string, patches []SpotFixPatch) *PatchResult {
    result := original
    appliedCount := 0
    for _, patch := range patches {
        // 检查 TARGET_TEXT 是否在原文中唯一出现
        count := strings.Count(result, patch.TargetText)
        if count == 1 {
            result = strings.Replace(result, patch.TargetText, patch.ReplacementText, 1)
            appliedCount++
        }
        // count == 0 → 未命中，跳过
        // count > 1 → 不唯一，跳过（避免误替换）
    }
    return &PatchResult{
        RevisedContent:     result,
        Applied:            appliedCount > 0,
        AppliedPatchCount:  appliedCount,
    }
}
```

```go
// ReviseChapter 执行章节修订
func (a *ReviserAgent) ReviseChapter(ctx context.Context, input *ReviseChapterInput) (*ReviseOutput, error) {
    // 1. 并行加载 10 个 truth files
    currentState, ledger, hooks, styleGuide, volumeOutline, storyBible,
    characterMatrix, chapterSummaries, parentCanon, fanficCanon :=
        parallelRead10Files(input.BookDir)

    bookRules, bookRulesBody := readBookRules(input.BookDir)

    // 3. 确定语言和 governed 模式

        input.Options.ContextPackage != nil &&
        input.Options.RuleStack != nil

        {Role: "user", Content: userPrompt},

        output.WordCount = countChapterLength(output.RevisedContent, input.Options.LengthSpec.CountingMode)

}

// buildTieredIssueList 分层展示问题（auto 模式）
func buildTieredIssueList(issues []AuditIssue, isEnglish bool) string {
    critical, high, medium := []string{}, []string{}, []string{}
    for _, issue := range issues {
        line := fmt.Sprintf("- %s: %s", issue.Category, issue.Description)
        switch issue.Severity {
        case "critical":
            critical = append(critical, line)
        case "warning":
            high = append(high, line)
        default:
            medium = append(medium, line)
        }
    }
    // 按 Critical → High → Medium 顺序拼接
    parts := []string{}
    if len(critical) > 0 {
        parts = append(parts, "## Critical（必须解决）\n"+strings.Join(critical, "\n"))
    }
    if len(high) > 0 {
        parts = append(parts, "## High（应当改善）\n"+strings.Join(high, "\n"))
    }
    if len(medium) > 0 {
        parts = append(parts, "## Medium（参考建议）\n"+strings.Join(medium, "\n"))
    }
    return strings.Join(parts, "\n\n")
}
```

## 附录：6 种模式对比总结

| 模式 | 修改幅度 | 输出格式 | 字数护栏 | 适用场景 |
|------|----------|----------|----------|----------|
| auto | 自动路由 | PATCHES 或 REVISED_CONTENT | 无（normalize 处理） | 审查循环默认调用 |
| polish | 最小 | REVISED_CONTENT | 有 | 文风/节奏微调 |
| rewrite | 中等 | REVISED_CONTENT | 有 | 段落重组/画面增强 |
| rework | 较大 | REVISED_CONTENT | 有 | 场景推进重构 |
| anti-detect | 中等 | REVISED_CONTENT | 有 | 降低 AI 痕迹 |
| spot-fix | 最小 | PATCHES | 有 | 零星句子修改 |

**auto 模式路由决策树**：

```
issues 为空？
├─ 是 → allow-full
└─ 否 → 有 repairScope 标注的阻塞问题？
   ├─ 是 → 有 structural？
   │  ├─ 是 → rewrite-only
   │  └─ 否 → 全部 local？ → patch-only / 否则 allow-full
   └─ 否 → 回退到 category 正则匹配
      ├─ 有结构类问题 → rewrite-only
      ├─ 全部局部类 → patch-only
      └─ 混合/未知 → allow-full
```
