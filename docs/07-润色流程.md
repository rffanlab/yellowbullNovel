## 一、流程概述

Polisher（润色器）是章节产出流水线的**最后一道文笔层工序**，在 reviewer + reviser 循环接受章节结构之后运行。它只触碰文字表面——句式、段落、用词、标点、五感、对话自然度——**绝不**改动情节、人设、主线。

```
Writer（写章）
   │
   ▼
Reviewer + Reviser 循环（结构/剧情问题修复）
   │
   ▼ 结构通过
   │
Polisher（文笔层润色）──► polishedContent
   │
   ▼
Normalize（字数归一，详见 06-回炉重写流程 的 normalize 阶段）
   │
   ▼
落盘 chapter_NN.md
```

Polisher 是**可选**工序：book.json 可配置是否启用。短篇流程默认不启用（短篇更看重一气呵成）。

## 二、Polisher 与 Reviser 的区别

inkos 同时存在两个看似相近但职责不同的"改文字"组件：

| 维度 | Polisher | Reviser（polish 模式） |
|------|----------|----------------------|
| **触发** | 独立工序，reviewer 接受后自动跑 | 由 issues 列表驱动（回炉或自动审查循环） |
| **输入** | 仅章节正文 + chapter memo | issues 列表 + 正文 + 上下文包 |
| **是否接收 issues** | ❌ 不接收 | ✅ 接收（auto 模式按 issue 路由） |
| **发现剧情问题** | 只标记 `[polisher-note]`，不动正文 | 直接按 issue 修改（rewrite/rework 模式可重构） |
| **改动边界** | 仅文字层（句式/用词/标点） | 按模式递增：polish < rewrite < rework |
| **温度** | 默认 0.4（稳定偏保守） | auto 模式按 issue 类型动态 |
| **输出契约** | 完整润色后正文 + 可选 polisher-note | REVISED_CONTENT 或 PATCH |
| **长度约束** | ±15% 原文 | 按模式不同（auto 委托给 normalize） |
| **定位** | "美化工"——结构定型后的最后抛光 | "修理工"——按问题清单修结构/剧情 |

### 2.1 为什么需要独立 Polisher

1. **Reviewer/Reviser 关注结构正确性**，不擅长也不该负责文笔细腻度。
2. **issues 驱动的 polish 模式**只能针对已知问题改，无法发现 reviewer 漏判的文笔雷点。
3. **Polisher 独立扫一遍**能消灭 reviewer 不报但读者难受的 6 类文笔雷点（见 §6）。
4. **职责隔离**：Polisher 不碰结构，避免在润色阶段意外破坏已通过审查的剧情连贯性。

### 2.2 何时用 Polisher、何时用 Reviser polish 模式

| 场景 | 选择 |
|------|------|
| 正常写章流水线，结构已通过 | **Polisher** |
| 回炉请求明确指定 polish 模式 + issues | **Reviser polish 模式** |
| 用户手动请求"只润色不改剧情"且无 issues | **Polisher** |
| 审查循环中检测到文笔类 issue | **Reviser auto/polish 模式** |

## 三、Polisher 的允许操作

Polisher 只能做以下三类操作，且必须保留原文绝大多数句子：

### 3.1 替换用词（同义/更精准）

- 把模糊词换成具体词：`走 → 踱` / `看 → 瞥` / `说 → 嘟囔`。
- 把 AI 标记词换成口语或动作：`不禁 → 删` / `仿佛 → 删` / `宛如 → 删`。
- 把叙述者结论换成具体感官：`他感到愤怒 → 他捏碎了茶杯`。
- 形容词降密度：一句话最多 1-2 个精准形容词，多余的删。

### 3.2 调整句序（节奏/呼吸感）

- 长短句交替：禁止连续 3 句以上同结构/同主语开头。
- 段落呼吸：3-5 行/段（手机阅读友好），连续 7 行以上必须拆段。
- 但不可把"动作+反应"拆碎到失去节奏。

### 3.3 修改标点节奏

- 删冗余破折号 `——`（存量出现一律改写）。
- 删 `……` 敷衍应答式标点堆砌。
- 用标点控制节奏：短句用句号断开，长句用逗号呼吸。
- 禁止 `不是……而是……` 句式（存量改写）。

## 四、Polisher 的禁止操作

以下操作硬禁止，违反即视为润色越界（reviewer 后续可检测到并回退）：

### 4.1 增删段落

- ❌ 不得新增整段（哪怕是为了补过渡）。
- ❌ 不得删除整段（哪怕看起来冗余）。
- ✅ 只能在段内改句子、拆段、合并相邻碎句。

### 4.2 改人名/地名/物品名

- ❌ 不得把"林晚"改成"林婉"。
- ❌ 不得把"青云山"改成"青冥山"。
- ❌ 不得把"碎星剑"改成"碎月剑"。
- ✅ 可以调整名字前后的修饰语，但专有名词本身不动。

### 4.3 增加新情节或新对话

- ❌ 不得给角色加原本没有的动作/台词。
- ❌ 不得插入新场景、新物件、新信息。
- ❌ 不得给对话加回合（哪怕只加一句"他说"）。
- ✅ 可以重写已有对话的措辞让其更自然，但语义边界不变。

### 4.4 改变因果关系

- ❌ 不得把"因为 A 所以 B"改成"B 之后才发生 A"。
- ❌ 不得把"甲先动手"改成"乙先动手"。
- ❌ 不得调整事件先后顺序。
- ✅ 可以重排句内分句顺序，但整体因果链不动。

### 4.5 长度约束

- 润色后总长变化不得超过原文 **±15%**。
- 超出则回退到原文（避免润色器借机大改）。

## 五、polisher-note 标记格式与处理方式

### 5.1 格式

Polisher 在润色过程中若发现**剧情/结构问题**（非文笔问题），不得在正文中修改，而是在正文最末尾另起一行，以 `[polisher-note] ` 开头写明，每条一行：

```
（正文最后一段结束）

[polisher-note] 第 3 段提到"碎星剑已断"，但第 7 章回收时它仍是完好状态，疑似连贯性问题，请 reviewer 复核。
[polisher-note] 主角对反派的称呼从"林兄"突然变成"林某"，未有人际转折，疑似人设漂移。
```

### 5.2 触发条件

polisher-note 只用于**剧情/结构类**问题，不用于文笔问题（文笔问题 Polisher 自己改）。典型场景：

| 问题类型 | 示例 |
|---------|------|
| 连贯性矛盾 | 前文说剑断，本章剑还在 |
| 人设漂移 | 称呼/语气突变无解释 |
| 主线偏移 | memo 要求推进 H007 但正文没碰 |
| memo 未兑现 | memo 的 goal 是"主角认出凶手"，正文没写 |
| 因果断裂 | 上一章 A 还没发生，本章直接引用 A 的结果 |

### 5.3 处理流程

```
Polisher 输出 polishedContent（含 polisher-note）
        │
        ▼
   落盘前剥离 polisher-note 行 ──► 纯正文写入 chapter_NN.md
        │
        ▼
   polisher-note 行写入 chapter_NN.polisher-notes.md（侧车文件）
        │
        ▼
   下一轮 reviewer 读取侧车文件 ──► 转为 AuditIssue 注入审查
        │
        ▼
   若有 critical issue ──► 触发回炉（06-回炉重写流程）
```

### 5.4 剥离规则

- 行首匹配 `[polisher-note]`（大小写不敏感）。
- 必须在正文最末尾连续出现，中间不得穿插正文。
- 剥离后若正文末尾出现空行，一并清理。

### 5.5 reviewer 读取

reviewer 在审查章节时，若 `chapter_NN.polisher-notes.md` 存在，把每行转为一个 `AuditIssue`：

```go
AuditIssue{
    Severity:   "warning", // 默认 warning，含"连贯/矛盾"关键词升级 critical
    Category:   "polisher-note",
    Description: noteText,
    Suggestion: "复核 polisher 标记的结构问题",
}
```

## 六、系统提示词设计

Polisher 的 system prompt 包含以下模块（中文版，英文版结构相同）：

### 6.1 边界声明（硬约束）

```
你只改文字层——句式 / 段落 / 排版 / 用词 / 五感 / 对话自然度。
你禁止增删情节、改变人设、调整主线。
发现情节/结构问题只能以 [polisher-note] 形式附在章末供下一轮 reviewer 参考，不能动正文。
结构的事归 Reviewer，不归你。
```

### 6.2 六条文笔雷点（要消灭的）

1. **描写无效**：冗长环境描写、与主线无关对话塞满页面 → 删到"一笔带过"。
2. **文笔华丽过度**：辞藻堆砌、形容词地毯轰炸 → 让文字服从情绪。
3. **文笔欠佳**：句意含混、指代不清、逻辑跳跃、语言干瘪 → 重写成通顺有画面感的句子。
4. **排版不规范**：段落过长、格式不统一、对话无换行 → 统一为手机阅读友好格式。
5. **AI 味痕迹**：转折词泛滥、"了"字堆砌、"仿佛/宛如/竟然"等情绪中介词、编剧旁白、分析报告式语言 → 替换成口语化表达或具体动作。
6. **群像脸谱化**：不写"众人齐声惊呼"，挑 1-2 个角色写具体反应。

### 6.3 文字层硬规约

- 段落 3-5 行/段，连续 7 行以上必须拆段。
- 句式多样化，禁止连续 3 句以上同结构/同主语开头。
- 动词 > 形容词，一句话最多 1-2 个精准形容词。
- 五感代入：场景至少 1-2 种感官细节，但不机械叠加。
- 对话自然度：角色有辨识度，符合身份/情绪/信息掌握。
- 情绪外化：`他感到愤怒 → 他捏碎了茶杯`。
- 删叙述者结论和 AI 标记词。
- 禁止破折号 `——`，禁止 `不是……而是……` 句式。

### 6.4 输出契约

```
直接返回润色后的完整章节正文——不要 JSON、不要章节标题行、不要任何解释。
如果发现必须交给 reviewer 的情节/结构问题，在正文末尾另起一行以 "[polisher-note] " 开头写明，每条一行。没有就不加。
保留原文绝大多数句子。只改真正有问题的句子，不要整段重写。
修改后章节总长变化不得超过原文字数 ±15%。
```

### 6.5 chapter memo 注入

若提供 chapterMemo，注入到 user prompt：

```
## 章节备忘（润色不得偏离此目标）
goal：{memo.goal}

{memo.body}
```

Polisher 用 memo 判断"正文是否偏离本章目标"，偏离则标 polisher-note，不自行修正。

## 七、流程时序

```
┌─────────────────────────────────────────────────────────┐
│ 1. 调用方组装 PolishChapterInput                         │
│    - chapterContent：reviewer 接受的正文                  │
│    - chapterNumber                                       │
│    - chapterMemo（可选）                                  │
│    - language（zh/en，默认 zh）                           │
│    - temperature（默认 0.4）                              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 2. 组装 system prompt（按 language）                      │
│    - 边界声明 + 6 雷点 + 硬规约 + 输出契约                 │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 3. 组装 user prompt                                       │
│    - "请润色第 N 章。只返回完整润色后正文..."              │
│    - memo block（若有）                                   │
│    - "## 待润色章节\n{chapterContent}"                    │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 4. 调用 LLM（temperature=0.4）                            │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 5. 后处理                                                 │
│    a. trim 响应                                          │
│    b. stripWrappingFence：去掉 ```...``` 包裹            │
│    c. 若结果为空 → 回退到原文                             │
│    d. 剥离 [polisher-note] 行 → 侧车文件                  │
│    e. 长度校验：变化 > ±15% → 回退到原文 + 标记 skipped   │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 6. 返回 PolishChapterOutput                              │
│    - polishedContent（纯正文）                            │
│    - changed（是否相对原文有改动）                         │
│    - polisherNotes（剥离出的 note 列表）                   │
│    - skipped（是否因长度回退）                             │
│    - tokenUsage                                           │
└─────────────────────────────────────────────────────────┘
```

## 八、完整 Go 接口定义

文件位置：`internal/agents/polisher.go`。

### 8.1 输入输出结构

```go
package agents

// PolishChapterInput 润色入参
type PolishChapterInput struct {
    ChapterContent string            `json:"chapterContent"` // 待润色正文
    ChapterNumber  int               `json:"chapterNumber"`
    ChapterMemo    *ChapterMemo      `json:"chapterMemo,omitempty"` // 可选：防漂移锚点
    Language       string            `json:"language,omitempty"`    // "zh"|"en"，默认 "zh"
    Temperature    float64           `json:"temperature,omitempty"` // 默认 0.4
}

// PolishChapterOutput 润色输出
type PolishChapterOutput struct {
    PolishedContent string      `json:"polishedContent"` // 纯正文（已剥离 polisher-note）
    Changed         bool        `json:"changed"`         // 是否相对原文有改动
    PolisherNotes   []string    `json:"polisherNotes,omitempty"` // 剥离出的结构问题标记
    Skipped         bool        `json:"skipped"`         // 是否因长度越界回退到原文
    TokenUsage      *TokenUsage `json:"tokenUsage,omitempty"`
}

// ChapterMemo 章节备忘（与 18-数据模型 一致）
type ChapterMemo struct {
    Goal string `json:"goal"`
    Body string `json:"body"`
}

// TokenUsage token 用量
type TokenUsage struct {
    PromptTokens     int `json:"promptTokens"`
    CompletionTokens int `json:"completionTokens"`
    TotalTokens      int `json:"totalTokens"`
}
```

### 8.2 PolisherAgent 接口

```go
package agents

// PolisherAgent 文笔层润色器
// 在 reviewer+reviser 循环接受章节结构后运行，只触碰文字表面。
// 发现剧情/结构问题只标记 [polisher-note]，不修改正文。
type PolisherAgent interface {
    BaseAgent

    // PolishChapter 润色一章
    //
    // 流程：
    //   1. 按 language 组装 system prompt（边界 + 6 雷点 + 硬规约 + 契约）
    //   2. 组装 user prompt（含可选 memo block）
    //   3. 调用 LLM（temperature 默认 0.4）
    //   4. stripWrappingFence 去掉代码块包裹
    //   5. 剥离 [polisher-note] 行到 PolisherNotes
    //   6. 长度变化 > ±15% 则 Skipped=true 并回退原文
    //
    // 返回 PolishChapterOutput，调用方负责落盘正文 + 侧车 notes 文件。
    PolishChapter(ctx context.Context, in PolishChapterInput) (PolishChapterOutput, error)
}

// NewPolisherAgent 构造默认 PolisherAgent
//   - llm: LLM 提供商（见 15-LLM提供商层）
//   - logger: 日志器
func NewPolisherAgent(llm LLMProvider, logger Logger) PolisherAgent
```

### 8.3 BaseAgent 嵌入接口

```go

    MaxTokens   int

    Usage   *TokenUsage
```

### 8.6 配置项

在 `book.json` 中配置：

```json
{
  "polisher": {
    "enabled": true,
    "temperature": 0.4,
    "language": "zh",
    "lengthBudget": 0.15,
    "injectMemo": true
  }
}
```

| 字段 | 默认 | 说明 |
|------|------|------|
| enabled | true | 是否在写章流水线末尾自动跑 Polisher |
| temperature | 0.4 | LLM 温度 |
| language | "zh" | 系统提示词语言 |
| lengthBudget | 0.15 | 允许长度变化比例（±15%） |
| injectMemo | true | 是否把 chapter memo 注入 prompt 防漂移 |

### 8.7 流水线集成

Polisher 在章节写作流水线中的调用位置（与 04-章节写作流程、05-章节审查循环流程 衔接）：

```go
// 简化伪代码
func (p *Pipeline) writeChapter(ctx context.Context, bookID string, num int) error {
    // 1. 写章
    draft := p.writer.Write(...)
    // 2. 审查 + 回炉循环
    accepted := p.reviewLoop(ctx, bookID, num, draft)
    // 3. 润色（若启用）
    if p.bookConfig.Polisher.Enabled {
        out, err := p.polisher.PolishChapter(ctx, PolishChapterInput{
            ChapterContent: accepted,
            ChapterNumber:  num,
            ChapterMemo:    memo,
            Language:       p.bookConfig.Polisher.Language,
            Temperature:    p.bookConfig.Polisher.Temperature,
        })
        if err != nil {
            return err
        }
        p.persister.SaveChapter(bookDir, num, out.PolishedContent)
        if len(out.PolisherNotes) > 0 {
            p.persister.SavePolisherNotes(bookDir, num, out.PolisherNotes)
        }
    } else {
        p.persister.SaveChapter(bookDir, num, accepted)
    }
    // 4. 字数归一（06-回炉流程 normalize 阶段）
    // 5. 结算 + 持久化 truth files
    return nil
}
```
