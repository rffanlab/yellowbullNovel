## 一、设计目标

| 目标 | 说明 |
|------|------|
| **功能对齐** | 覆盖 inkos 核心写作链路：建书 → 规划 → 组装 → 写作 → 审查 → 结算 → 持久化 |
| **多数据库支持** | 通过 GORM 统一抽象，支持 SQLite（开发）/ MySQL / PostgreSQL（生产）切换 |
| **LLM 提供商可插拔** | 统一 LLM provider 接口，支持 OpenAI / Anthropic / 自定义端点 |
| **前后端分离** | Go 后端提供 HTTP API，Vue3 前端独立部署，可独立升级 |
| **并发安全** | 保留 inkos 的书籍写锁机制，支持多书并发写作 |
| **可观测** | Token 用量、审计分数、字数遥测等指标全程记录 |

## 三、技术栈

### 后端（Go）

| 组件 | 选型 | 用途 |
|------|------|------|
| Web 框架 | **Gin** | HTTP API 路由、中间件、请求绑定 |
| ORM | **GORM** | 多数据库抽象（SQLite/MySQL/PostgreSQL） |
| 配置 | **Viper** | YAML / ENV / flags 多源配置 |
| 日志 | **Zap** | 结构化日志 |
| 调度 | **robfig/cron** | 定时写作任务 |
| LLM SDK | 自实现 | 统一 provider 接口，对接 OpenAI/Anthropic/自定义 |
| 测试 | **testify** | 断言 + mock |

### 前端（Vue3）

| 组件 | 选型 | 用途 |
|------|------|------|
| 框架 | **Vue 3** | Composition API |
| 构建 | **Vite** | 开发服务器 + 打包 |
| 语言 | **TypeScript** | 类型安全 |
| 状态 | **Pinia** | 全局状态管理 |
| UI 库 | Element Plus / Naive UI | 组件库 |
| HTTP | **Axios** | API 调用 |
| 路由 | **Vue Router** | 前端路由 |

### 数据库

| 数据库 | 场景 | 驱动 |
|--------|------|------|
| SQLite | 开发/单机部署 | `gorm.io/driver/sqlite` |
| MySQL | 生产/中小规模 | `gorm.io/driver/mysql` |
| PostgreSQL | 生产/大规模 | `gorm.io/driver/postgres` |

通过配置项 `database.driver` 切换，GORM 统一抽象。

### 请求流示例（写下一章）

```
前端 POST /api/books/:id/write-next
  → api.WriteNextChapter handler
    → pipeline.PipelineRunner.WriteNextChapter(bookID)
      → planner.PlanChapter()          → llm.Chat()  生成 Intent + Memo
      → composer.ComposeChapter()      → 组装 ContextPackage
      → writer.WriteChapter()          → llm.Stream() 生成正文
      → settler.Settle()               → llm.Chat()  状态结算
      → auditor.AuditChapter()         → llm.Chat()  33维审查
      → [未通过] reviser.Revise()      → llm.Chat()  修订
      → state.ValidateAndPersist()     → 写文件 + DB
    → 返回 ChapterPipelineResult
  → JSON 响应
```

## 五、模块依赖关系

```
api  ──────────────►  pipeline  ──────────────►  agents
 │                      │                          │
 │                      │                          ▼
 │                      ▼                       state
 │                    state                        │
 │                      │                          ▼
 │                      ▼                        llm
 │                    hooks                       │
 │                      │                          ▼
 ▼                      ▼                       storage
config ◄──────────────────────────────────────────┘
  │
  ▼
storage
```

依赖方向说明：

| 包 | 依赖 | 被依赖 |
|----|------|--------|
| `api` | pipeline, config, models | cmd/server |
| `pipeline` | agents, state, hooks, config, models | api, scheduler |
| `agents` | state, llm, hooks, models | pipeline |
| `state` | llm(仅类型), storage, models | agents, pipeline |
| `hooks` | models, storage | agents, pipeline, state |
| `scheduler` | pipeline, config | cmd/scheduler |
| `llm` | config, models | agents |
| `storage` | config, models | state, hooks, pipeline |
| `config` | models | 所有包 |
| `models` | (无) | 所有包 |

**关键原则**：
- `models` 是最底层包，不依赖任何 internal 包
- `llm` 只被 `agents` 依赖，不反向依赖
- `pipeline` 编排 `agents`，不直接调用 `llm`
- 所有包通过接口依赖，不依赖具体实现

## 八、Go 包结构设计

### 8.1 `internal/models/` - 数据模型

纯数据结构定义，不依赖任何 internal 包。对应 inkos 的 `models/*.ts`。

```
internal/models/
├── book.go              # BookConfig, BookStatus, Platform, FanficMode
├── chapter.go           # ChapterMeta, ChapterStatus
├── input_governance.go  # ChapterIntent, ChapterMemo, ContextPackage, RuleStack, ChapterTrace
├── runtime_state.go     # RuntimeStateDelta, HookRecord, HookStatus, HookOps
├── length_governance.go # LengthSpec, LengthTelemetry, LengthCountingMode
├── project.go           # ProjectConfig, LLMConfig, NotifyConfig, DetectionConfig
├── genre_profile.go     # GenreProfile
├── book_rules.go        # BookRules
├── detection.go         # DetectionHistoryEntry, DetectionStats
└── audit.go             # AuditResult, AuditIssue
```

详细结构体定义见 [18-数据模型.md](./18-数据模型.md)。

### 8.2 `internal/llm/` - LLM 提供商层

统一 LLM 调用接口，支持多提供商切换。对应 inkos 的 `llm/`。

```
internal/llm/
├── provider.go          # Provider 接口定义 + LLMClient/LLMResponse/LLMMessage
├── client.go            # LLMClient 工厂（createLLMClient）
├── stream.go            # 流式响应处理 + StreamMonitor
├── secrets.go           # 密钥管理（从 env/config 读取）
├── model_lookup.go      # 模型卡查询（maxOutput/contextWindow）
├── service_presets.go   # 服务预设（已知端点配置）
├── think_tag.go         # think 标签剥离
└── providers/
    ├── openai.go        # OpenAI 端点
    ├── anthropic.go     # Anthropic 端点
    ├── custom.go        # 自定义端点
    └── registry.go      # 端点注册表
```

核心接口：

```go
// Provider 定义 LLM 调用的统一接口
type Provider interface {
    // Chat 同步对话
    Chat(ctx context.Context, messages []LLMMessage, opts ...Option) (*LLMResponse, error)
    // Stream 流式对话
    Stream(ctx context.Context, messages []LLMMessage, onChunk func(string), opts ...Option) (*LLMResponse, error)
}
```

### 8.3 `internal/state/` - 状态管理

书籍状态、锁、记忆库、状态增量应用。对应 inkos 的 `state/`。

```
internal/state/
├── manager.go           # StateManager — 书籍锁/配置/truth files 读写
├── memory_db.go         # MemoryDB — SQLite 记忆检索
├── runtime_store.go     # RuntimeStateStore — 运行时状态快照
├── reducer.go           # StateReducer — 应用 RuntimeStateDelta
├── bootstrap.go         # StateBootstrap — 从 Markdown 重建状态
├── projections.go       # 状态投影（派生视图）
└── validator.go         # 状态校验器
```

核心职责：

| 组件 | 职责 |
|------|------|
| StateManager | 书籍写锁（进程级，心跳续约）、配置读写、truth files 读写 |
| MemoryDB | SQLite 记忆库，支持 facts/summaries/hooks/volume_summaries 语义检索 |
| RuntimeStateStore | 每章写完后保存状态快照，支持回滚 |
| StateReducer | 将 Settler 产出的 RuntimeStateDelta 应用到 truth files |
| StateBootstrap | 从 Markdown 文件重建结构化状态（导入续写场景） |
