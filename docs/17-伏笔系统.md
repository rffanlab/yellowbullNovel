## 一、系统概述

伏笔系统是 inkos 写作流水线的"债务账本"。每埋下一个伏笔就是向读者借一笔债，每推进/回收一笔就是还债。伏笔系统的核心职责：

1. **准入**：拒绝无意义、重复的伏笔进入活跃池。
2. **跟踪**：记录每条伏笔的种植章、最近推进章、预期回收时机。
3. **诊断**：标记陈旧（stale）、受阻（blocked）、逾期（overdue）、可回收（readyToResolve）。
4. **晋升**：把 architect 产出的种子（seed）按规则提升为活跃账本条目。
5. **仲裁**：本章 RuntimeStateDelta 中声明的新伏笔/推进/回收是否合规。
6. **健康度**：评估整个伏笔池是否膨胀、是否长期不动。
7. **预算**：活跃伏笔数上限 12，接近上限时向 planner 发出警告。
8. **账本校验**：验证章节正文是否兑现了 memo 中承诺的伏笔操作。

```
Architect 产出 seed 池
     │
     ▼
hook-promotion（晋升判定）──► promoted=true 进入 pending_hooks.md 账本
     │
     ▼
Writer 写章 ──► RuntimeStateDelta（hookOps: upsert/mention/resolve/defer）
     │
     ▼
hook-arbiter（仲裁 delta 中的 hook 操作）──► 合规的 delta
     │
     ▼
hook-governance / hook-health（健康度 + 陈旧检测）──► AuditIssue 列表
     │
     ▼
hook-ledger-validator（正文 vs memo 承诺）──► 违规列表
```

## 二、伏笔生命周期

伏笔从被 architect 播种到最终回收，经历以下阶段：

```
seed（种子）─► active（活跃）─► advanced（推进）─► resolved（回收）
                                      │
                                      └─► deferred（延后）─► active（重新激活）
```

### 2.1 seed 种子

- 由 architect 在建书骨架阶段产出，存在于 `pending_hooks.md` 的种子池。
- `startChapter = 0`（未真正落到正文章节）。
- `promoted = false`，不进入活跃账本，不参与预算计数。
- 种子是对"将来要埋的伏笔"的预案，writer 不必在本章处理。

### 2.2 active 活跃

- 种子被晋升（`promoted = true`）或在正文中首次落地后进入活跃态。
- `startChapter > 0`，`status = "open"`。
- 计入活跃伏笔预算（上限 12）。
- reviewer / planner / writer 都能在上下文里看到它。

### 2.3 advanced 推进

- 伏笔在本章被"真实推进"：信息变化、冲突升级、角色行动触及该伏笔。
- `lastAdvancedChapter` 更新为当前章号。
- `status` 可保持 `open` 或转为 `progressing`。
- 纯粹重述（restatement）不算推进，只算 `mention`。

### 2.4 resolved 回收

- 伏笔的预期回报（expectedPayoff）在本章兑现。
- `status = "resolved"`，退出活跃池，不再计入预算。
- 回收时通常需要配套开新伏笔（"揭 1 埋 1"硬底线，见 §9）。

### 2.5 deferred 延后

- 伏笔暂时不处理，但保留在账本中。
- `status = "deferred"`，退出活跃池（不参与预算），但不出账本。
- 后续可重新激活回到 active。

### 2.6 生命周期描述器 HookLifecycle

`describeHookLifecycle` 根据当前章号、目标章数、payoffTiming 计算每条伏笔的状态指标：

| 指标 | 含义 |
|------|------|
| `timing` | 归一化后的回收时机（immediate/near-term/mid-arc/slow-burn/endgame） |
| `phase` | 当前故事阶段（opening/middle/late），按章号或进度比推断 |
| `age` | 伏笔年龄 = 当前章 - 种植章 |
| `dormancy` | 休眠章数 = 当前章 - max(种植章, 最近推进章) |
| `readyToResolve` | 是否进入可回收窗口 |
| `stale` | 是否陈旧（休眠超阈值或逾期且无动量） |
| `overdue` | 是否逾期（年龄超 overdueAge） |
| `advancePressure` | 推进压力分（age + dormancy + 陈旧/逾期加成） |
| `resolvePressure` | 回收压力分（resolveBias × 系数 + 各类加成） |

### 2.7 时机档案 HOOK_TIMING_PROFILES

不同 payoffTiming 对应不同的生命周期阈值：

| timing | earliestResolveAge | staleDormancy | overdueAge | minimumPhase | resolveBias |
|--------|-------------------|---------------|------------|--------------|-------------|
| immediate | 1 | 1 | 3 | opening | 5 |
| near-term | 1 | 2 | 5 | opening | 4 |
| mid-arc | 2 | 4 | 8 | opening | 3 |
| slow-burn | 4 | 5 | 12 | middle | 2 |
| endgame | 6 | 6 | 16 | late | 1 |

### 2.8 阶段阈值

- 按 progress（当前章 / 目标章）划分：`< 0.33` opening，`[0.33, 0.72)` middle，`>= 0.72` late。
- 无目标章数时按绝对章号：`< 8` opening，`[8, 24)` middle，`>= 24` late。

## 三、HookRecord 13 列详解

HookRecord 是伏笔账本（`pending_hooks.md` 表格 + SQLite `stored_hooks` 表）的每一行。共 13 列：

| # | 列名 | 类型 | 必填 | 说明 |
|---|------|------|------|------|
| 1 | `HookID` | string | 是 | 伏笔唯一标识，由 `buildCanonicalHookId` 生成（slug 化 type+payoff+notes），冲突时追加 `-2`/`-3` 后缀。 |
| 2 | `Type` | string | 是 | 伏笔类型描述，如"身世之谜"、"信物线索"、"承诺回收"。准入时不能为空。 |
| 3 | `Status` | enum | 是 | `open` / `progressing` / `deferred` / `resolved`。归一化函数 `normalizeStoredHookStatus` 兼容中文别名（已回收/延后/持续推进等）。 |
| 4 | `StartChapter` | int | 是 | 种植章号。`0` 表示仍是 architect 种子未落地；`>0` 表示已在正文中埋下。 |
| 5 | `LastAdvancedChapter` | int | 是 | 最近一次"真实推进"的章号。初始等于 `StartChapter`。纯 mention 不更新此字段。 |
| 6 | `ExpectedPayoff` | string | 否 | 预期回报描述，如"第 20 章揭示真相"。准入时与 notes 合并作为 payoffSignal，为空则拒绝准入。 |
| 7 | `PayoffTiming` | enum | 否 | `immediate` / `near-term` / `mid-arc` / `slow-burn` / `endgame`。未显式给出时由 `inferHookPayoffTiming` 从 expectedPayoff+notes 文本推断。 |
| 8 | `Notes` | string | 否 | 自由备注，补充伏笔细节、关联角色、回收条件等。 |
| 9 | `DependsOn` | []string | 否 | 上游因果依赖（其他 HookID 列表）。上游未种植或未回收时本伏笔"受阻(blocked)"。 |
| 10 | `PaysOffInArc` | string | 否 | 预期回收的卷/弧描述，如"第 2 卷"。用于跨卷判定。 |
| 11 | `CoreHook` | bool | 否 | 是否核心主线伏笔。`true` 时强制晋升（promoted=true），reviewer 对其陈旧会升级严重度。 |
| 12 | `HalfLife` | int | 否 | 半衰期（章数）。未显式设置时按 timing 默认：immediate/near-term=10，mid-arc=30，slow-burn/endgame=80。 |
| 13 | `Promoted` | bool | 否 | 是否已从种子晋升为活跃账本条目。`false`/`nil` 时不计入预算，reviewer 不对其报 critical。 |

> 第 9-13 列为 Phase 7 引入的因果/晋升元数据，全部可选以兼容旧版 8 列账本。

### 3.1 advancedCount（派生字段，非账本列）

`advancedCount` 不在 13 列账本中，而是从 `chapter_summaries.md` 的 hookActivity 列派生统计：某 HookID 在多少个章节摘要的"伏笔动态"列中被提及。用于晋升判定（`advancedCount >= 2` 触发晋升）。

## 四、伏笔治理 hook-governance

### 4.1 准入控制

`evaluateHookAdmission` 决定一个候选伏笔是否允许进入活跃池。

**输入**：`HookAdmissionCandidate`（type + expectedPayoff + payoffTiming + notes）+ 当前活跃伏笔列表。

**校验顺序**：

1. **type 非空**：`normalizeText(type)` 为空 → 拒绝，reason=`missing_type`。
2. **payoffSignal 非空**：expectedPayoff + notes 合并后为空 → 拒绝，reason=`missing_payoff_signal`。
3. **重复家族检测**：与每条活跃伏笔比较
   - 完全归一化文本相同 → 拒绝，reason=`duplicate_family`，matchedHookId=对方ID。
   - type 相同且（英文词重叠 ≥2 或 中文 bigram 重叠 ≥3）→ 拒绝，reason=`duplicate_family`。
4. 全部通过 → admit=true，reason=`admit`。

**归一化**：`trim + lowercase + 仅保留 [a-z0-9\u4e00-\u9fff] + 压缩空白`。

**停用词**：`that/this/with/from/into/still/just/have/will/reveal` 等不参与重叠计数。

### 4.2 分类：core_hook vs 普通伏笔

| 维度 | core_hook=true | 普通伏笔 |
|------|----------------|----------|
| 晋升 | 强制 promoted=true（无需等 advancedCount） | 需满足晋升条件之一 |
| 预算 | 计入活跃上限 12 | 计入活跃上限 12 |
| 陈旧严重度 | reviewer 升级为 critical | warning |
| 账本可见 | 始终在 pending_hooks.md | 仅 promoted 后进入 |
| 上下文选取 | 优先注入 writer/planner 上下文 | 按压力分排序选取 |

### 4.3 陈旧检测 hook-stale-detection

`computeHookDiagnostics` 为每条伏笔计算诊断标志。**纯函数，无副作用**——结果用于渲染 markdown 标记和 reviewer 判定。

**输入**：全部伏笔列表 + 当前章号。

**输出**：`Map<HookID, HookDiagnostics>`。

```go
type HookDiagnostics struct {
    Stale           bool     // 过期：已种植 + 非resolved + distance > halfLife
    Blocked         bool     // 受阻：dependsOn 中存在未种植或未回收的上游
    MissingUpstream []string // 受阻的上游 HookID 列表
    Distance        int      // 距离 = 当前章 - 种植章
    HalfLife        int      // 半衰期（章数）
    BlockedDistance int      // 受阻持续章数 = 当前章 - 最早未清上游的参考章
}
```

**stale 判定**：
- 非已回收状态
- 已真正种植（`startChapter > 0`，纯种子不算 stale）
- `distance > halfLife`

**blocked 判定**：
- dependsOn 中任一上游"未种植"（startChapter=0 或 > 当前章）或"未回收"
- 本伏笔自身也非已回收

**blockedDistance 计算**（Phase 7 hotfix 3）：
- 对每个未清上游，参考章 = 上游已种植则用上游.startChapter，否则用本伏笔.startChapter。
- blockedDistance = 当前章 - min(所有参考章)。
- reviewer 据此判定：blocked ≥5 章触发 warning，≥6 章升级。

**渲染标记**：
- stale → `过期 (距=N/半衰=M)`
- blocked → `受阻于 H007, H012 (已阻 Y 章)`

## 五、伏笔晋升 hook-promotion

`shouldPromoteHook` 决定一条种子是否晋升为活跃账本条目。满足以下**任一**条件即晋升：

| 条件 | reason | 说明 |
|------|--------|------|
| `coreHook == true` | `core_hook` | 主线承重伏笔，恒在账本 |
| `len(dependsOn) > 0` | `depends_on` | 有上游因果依赖，值得跟踪 |
| `advancedCount >= 2` | `advanced_count` | 已在 ≥2 章被推进，证明读者已在跟踪 |
| 跨卷 | `cross_volume` | 跨越卷边界（见下） |

**跨卷判定 cross_volume**（需 volumeBoundaries ≥2 卷）：

- Case A：上游依赖声明在更晚的卷。
- Case B：`paysOffInArc` 明确提到不同卷（正则匹配 `第 N 卷` / `volume N` / `vol. N`）。
- Case C：payoffTiming 为 `endgame`/`slow-burn` 且种子在非末卷。

**默认半衰期**：

```go
func defaultHalfLifeChapters(timing HookPayoffTiming) int {
    switch timing {
    case PayoffImmediate, PayoffNearTerm:
        return 10
    case PayoffSlowBurn, PayoffEndgame:
        return 80
    default: // mid-arc 或未知
        return 30
    }
}
```

**轻量晋升重跑 rerunPromotionPass**：

consolidator 在每章后调用，从 `chapter_summaries.md` 派生 advancedCount，把 `advancedCount >= 2` 的未晋升种子翻转为 `promoted=true`。无 LLM 调用，纯文本统计。

## 六、伏笔健康分析 hook-health

`analyzeHookHealth` 评估整个伏笔池的健康度，产出 `AuditIssue` 列表（warning 级）。

### 6.1 健康度阈值默认值

```go
const (
    DefaultMaxActiveHooks       = 12  // 活跃伏笔上限
    DefaultStaleAfterChapters   = 10  // N 章未推进算陈旧
    DefaultNoAdvanceWindow      = 5   // 连续 N 章无真实推进
    DefaultNewHookBurstThreshold = 2   // 新开伏笔突发阈值
)
```

### 6.2 检查项

| 检查 | 触发条件 | 建议 |
|------|----------|------|
| **活跃超限** | activeHooks > 12 | 优先推进/回收/延后已有伏笔，再开新伏笔 |
| **压力未解** | 存在 stale/overdue/readyToResolve 伏笔且本章未处理（disposition=none/mention） | 让一个压力区伏笔发生真实推进/回收/明确延后 |
| **长期停滞** | 无压力伏笔但连续 ≥5 章无真实推进 | 下一章优先让旧伏笔真实推进 |
| **新开突发** | 本章新开 ≥2 个伏笔且未回收任何旧债 | 新开伏笔时配套回收旧伏笔 |

### 6.3 disposition 分类

`classifyHookDisposition` 根据本章 delta 判定每条伏笔的处置：

| disposition | 条件 |
|-------------|------|
| `defer` | hookId ∈ delta.hookOps.defer |
| `resolve` | hookId ∈ delta.hookOps.resolve |
| `advance` | delta.hookOps.upsert 中存在该 hookId 且 lastAdvancedChapter = 当前章 |
| `mention` | hookId ∈ delta.hookOps.mention |
| `none` | 以上都不满足 |

### 6.4 压力描述渲染

压力区伏笔最多列 3 条，格式：`H007（中程，可回收）、H012（慢烧，已逾期），另有 2 条`。

## 七、伏笔仲裁 hook-arbiter

`arbitrateRuntimeStateDeltaHooks` 在状态归约前对本章 delta 中的 hook 操作做合规仲裁。

### 7.1 输入输出

- **输入**：当前全部 hooks + 本章 RuntimeStateDelta（含 hookOps.upsert/mention/resolve/defer + newHookCandidates）。
- **输出**：resolvedDelta（合规化后的 delta）+ decisions（每条候选的处置决策）。

### 7.2 仲裁流程

```
1. 已知 HookID 的 upsert ──► 直接替换 workingHooks
2. 未知 HookID 的 upsert ──► 加入 fallbackCandidates（带 preferredHookId）
3. 对 [fallbackCandidates + newHookCandidates] 逐条评估准入:
   a. admit=true ──► createCanonicalHook，action=created
   b. duplicate_family + matchedHookId:
      - isPureRestatement ──► mention matchedHookId，action=mentioned
      - 有新信息 ──► mergeCandidateIntoExistingHook，action=mapped
   c. 其它拒绝 ──► action=rejected
4. 清理 mention 列表：去掉已被 upsert/resolve/defer 的
5. 返回 resolvedDelta（newHookCandidates 清空，upsert 已合并）
```

### 7.3 决策类型

```go
type HookArbiterAction string
const (
    ArbiterCreated  HookArbiterAction = "created"   // 新建伏笔
    ArbiterMapped   HookArbiterAction = "mapped"    // 合并到已有伏笔
    ArbiterMentioned HookArbiterAction = "mentioned" // 仅提及（纯重述）
    ArbiterRejected HookArbiterAction = "rejected"  // 拒绝
)
```

### 7.4 合并策略 mergeCandidateIntoExistingHook

- `type` / `expectedPayoff` / `notes`：取更丰富的文本（`preferRicherText`，长度长者胜）。
- `status`：已回收保持已回收，否则转为 `progressing`。
- `lastAdvancedChapter`：取 max(原值, 当前章)。
- `payoffTiming`：用 `resolveHookPayoffTiming` 重新解析。

### 7.5 纯重述判定 isPureRestatement

候选与已有伏笔的归一化文本完全相同，或候选的所有 term/bigram 都是已有伏笔的子集（新词数=0 且新 bigram <2），视为纯重述——只 mention 不合并。

### 7.6 HookID 生成 buildCanonicalHookId

1. 若候选带 preferredHookId 且不冲突 → 直接用。
2. 否则 `slugifyHookStem(type + expectedPayoff + notes)`：取前 5 个英文词 + 前 3 个中文段，连字符拼接，截断 64 字符。
3. 冲突时追加 `-2`/`-3` 后缀。

## 八、伏笔预算

### 8.1 上限

活跃伏笔（status != resolved/deferred 且 promoted != false）上限 **12 条**。

### 8.2 接近上限警告

当活跃数 ≥ 10（即 ≥ 上限的 5/6）时，planner 收到上下文警告：

> 当前有 N 个活跃伏笔，接近上限 12 个。优先推进、回收或延后已有伏笔，再继续开新伏笔。

### 8.3 超限硬处理

- 超过 12 时 `analyzeHookHealth` 产出 warning AuditIssue。
- hook-arbiter 仍按准入规则放行，但 planner 会在下一章规划时优先安排回收/推进。
- 不做硬拒绝（避免 writer 无伏笔可写时卡死），但 reviewer 会把超限标记为高风险。

### 8.4 揭 1 埋 1 软约束

- 每回收 1 个伏笔，至少新开 1 个相关新伏笔（见 §9 硬校验）。
- 软推荐"揭 1 埋 2"但非硬门（避免与"每章新开 ≤2"上限冲突）。

## 九、Hook Ledger 校验

`validateHookLedger` 是一道硬门：章节正文必须兑现 memo 中"## 本章 hook 账"承诺的伏笔操作。

### 9.1 memo 中的 hook 账格式

planner 在 chapter memo 中写：

```markdown
## 本章 hook 账

open:
- [new] 身世信物在黑市出现
- H021 "胖虎借条" 复活

advance:
- H007 "胖虎借条" → 主角决定还钱

resolve:
- H005 "初登场承诺" → 兑现

defer:
- H012
```

### 9.2 解析 parseHookLedger

- 定位 `## 本章 hook 账` / `## Hook ledger for this chapter` 标题段。
- 按 `open:` / `advance:` / `resolve:` / `defer:` 子段切换。
- 每行 `- ...` 解析出 HookID + descriptor + keywords。
- `[new]` 占位行计入 `newOpenCount` 但不进 entries（无 ID 可匹配）。
- 占位词（无/空/none/n/a/tbd 等）跳过。

### 9.3 关键词提取

从 descriptor 提取正文匹配关键词：

1. **优先**：引号内文本 `"..."` 或 `"..."`。
2. **回退**：箭头 `→` / `->` 之前的文本。
3. CJK 连续 ≥2 字符的整段 + 滑动 2-gram（长度 ≥3 时）+ 首/尾 3-gram（长度 ≥4 时）。
4. ASCII ≥3 字母词（小写化，过滤停用词）。

### 9.4 证据检查 draftEchoesEntry

对 advance + resolve 段的每条 entry，检查正文是否包含至少 1 个 keyword：

- ASCII keyword：`strings.Contains(strings.ToLower(draft), kw)`。
- CJK keyword：`strings.Contains(draft, kw)`。
- 无 keyword 的裸 ID：回退到 `\bID\b` 正则匹配。

**未找到证据** → warning 违规：

> memo 在 advance/resolve 里声明要处理 H007，但确定性关键词检查没有找到对应落点。复核正文是否已用动作/对话/物件或信息变化推进了 H007。

### 9.5 揭 1 埋 1 硬底线

当 `resolveCount > 0` 且 `openedCount < resolveCount` 时 → **critical** 违规：

> 本章 resolve 了 N 个钩子，但 open 只有 M 个新钩子。只揭不埋会让读者豁然开朗后索然无味。

- `openedCount = open.entries.length + newOpenCount`。
- 不校验 open/defer 段（新伏笔无 ID 可匹配；defer 是有意不处理）。
