## 一、流程概述

建书骨架是写作流水线的第一阶段，产出整本书的**散文密度基础设定**。流程由两个 Agent 协作完成：

```
BookConfig + externalContext
        │
        ▼
 ┌───────────────┐
 │ ArchitectAgent │ ── LLM(temperature=0.8) ──► 5 个 SECTION 原文
 │ generateFoundation                         │
 └───────┬───────┘
         │ parseSectionsWithRepair（解析失败自动修复重试）
         ▼
  ArchitectOutput { storyFrame, volumeMap, roles, bookRules, pendingHooks }
         │
         ▼
 ┌─────────────────────────┐
 │ FoundationReviewerAgent  │ ── LLM(temperature=0.3) ──► 5 维度评分
 │ review                   │
 └───────┬─────────────────┘
         │
    totalScore >= 80 且 单维度 >= 60 ?
         │
    ├── 否（重试 < 2 次）──► ArchitectAgent.generateFoundation(reviewFeedback) 循环
    ├── 是 ──► writeFoundationFiles 落盘
    └── 重试耗尽 ──► 返回最优结果 + 警告
```

核心设计原则（来自 inkos Phase 5 consolidation）：

1. **散文密度**：story_frame / volume_map / roles 必须是散文段落，不是表格、不是 bullet list。这本书的"灵气"从架构师的散文密度开始。
2. **去重铁律**：主角弧线只在 `roles` 写；世界铁律只在 `story_frame` 写；节奏原则只在 `volume_map` 尾段写；角色当前现状只在 `roles` 写；初始钩子只在 `pending_hooks`（startChapter=0 行）写。
3. **5 段 SECTION 合约**：从 inkos v13 的 7 段合并为 5 段（删除了独立的 `rhythm_principles` 和 `current_state`），输出体积缩小 25-40%。
4. **一人一卡**：主要角色和次要角色各自独立文件，主角卡是角色弧线的唯一权威来源。

## 三、ArchitectAgent 设计

### 3.1 输入

ArchitectAgent 的 `generateFoundation` 方法接收三个输入：

| 参数 | 类型 | 说明 |
|------|------|------|
| `book` | `*BookConfig` | 书籍元信息：platform / genre / title / targetChapters / chapterWordCount / language |
| `externalContext` | `string` | 来自外部系统的创作指令，会被织入 prompt 的「外部指令」段 |
| `reviewFeedback` | `string` | 上一轮审查反馈，触发「上一轮审核反馈」段，要求 LLM 明确修复而非换措辞 |

此外还有一个可选的 `reviseFrom` 结构，用于**既有架构稿修订模式**（把已有书的条目式架构稿升级为段落式）：

```go
type ReviseFrom struct {
    StoryBible       string // 原 story_bible 全文
    VolumeOutline    string // 原 volume_outline 全文
    BookRules        string // 原 book_rules 全文
    CharacterMatrix  string // 原 character_matrix 全文
    UserFeedback     string // 用户额外要求
}
```

### 3.2 LLM 输出契约：5 个 SECTION

LLM 必须按以下格式输出，用 `=== SECTION: xxx ===` 分块，严格顺序：

```
=== SECTION: story_frame ===
（4 段散文：主题与基调 / 核心冲突+前台后台双层故事 / 世界观底色 / 终局方向+全书Objective）

=== SECTION: volume_map ===
（5 段主体 + 1 段节奏原则尾段：各卷主题与情绪曲线 / 卷间钩子与回收承诺 / 各卷OKR / 卷尾必须发生的改变 / 节奏原则）

=== SECTION: roles ===
---ROLE---
tier: major
name: 林辞
---CONTENT---
（散文角色卡：核心标签 / 反差细节 / 人物小传 / 主角弧线 / 当前现状 / 关系网络 / 内在驱动 / 成长弧光）
---ROLE---
tier: minor
name: 张三
---CONTENT---
（简化角色卡：核心标签 / 反差细节 / 当前现状 / 与主角关系）

=== SECTION: book_rules ===
（普通 Markdown 规则卡，不写 YAML/JSON/代码块）

=== SECTION: pending_hooks ===
（13 列 Markdown 表格）
```

**去重铁律**（prompt 里必须强调）：
- 主角弧线只写在 `roles`，`story_frame` 只放一句指针
- 世界铁律只写在 `story_frame.世界观底色`，不在 `book_rules` 重复
- 节奏原则只写在 `volume_map` 尾段，不作为独立 section
- 角色当前现状只写在 `roles.当前现状`
- 初始钩子只写在 `pending_hooks`（startChapter=0 行）
- 环境/时代锚（仅历史/年代题材需要）织进 `story_frame.世界观底色`

### 3.3 各 SECTION 内容要求与字数预算

| SECTION | 字数预算 | 内容要求 |
|---------|---------|----------|
| `story_frame` | ≤ 3000 chars | 4 段散文，每段约 600-900 字。段1：主题与基调（具体命题+基调+指向主角的指针）；段2：核心冲突+对手定性+**前台/后台双层故事**（必须有因果关联）；段3：世界观底色（3-5条铁律 prose + 感官质感锚 + 叙事视角/专属规则）；段4：终局方向（最后一个镜头）+ **全书 Objective 一句话**（可验证的终局状态） |
| `volume_map` | ≤ 5000 chars | 5 段主体 + 1 段节奏原则尾段。只写到**卷级 prose**，禁止指定具体章号。段1：各卷主题与情绪曲线；段2：卷间钩子与回收承诺（**覆盖前台+后台两层**）；段3：各卷 OKR（每卷 1 个 Objective + 3 个 Key Results，KR 是下游 planner 排章的直接依据）；段4：卷尾必须发生的改变（不可逆事件）；段5：6 条节奏原则（至少 3 条具体化到本书） |
| `roles` | 总 ≤ 8000 chars | 一人一卡。主要角色至少 3 个（主角+主要对手+主要协作者），建议 2-3 主 + 2-3 辅。主角卡必须含：核心标签/反差细节/人物小传/**主角弧线(起点→终点→代价)**/当前现状/关系网络/内在驱动/成长弧光。次要角色简化版（4 个小标题），3-5 个 |
| `book_rules` | ≤ 1000 chars | 普通 Markdown 规则卡。只写给运行时和写手都能读懂的可执行规则，不写长篇散文。包含：主角（名字+性格锁+行为约束）/题材锁/叙事人称/数值资源规则(如适用)/年代限制(如适用)/禁止事项 |
| `pending_hooks` | ≤ 2000 chars | 13 列 Markdown 表格。可含 startChapter=0 种子行。建书阶段 lastAdvancedChapter 统一填 0。普通种子行状态写「暂缓」非「open」 |

**pending_hooks 表格 13 列**：

| 列号 | 列名 | 说明 |
|------|------|------|
| 1 | hook_id | 伏笔 ID，如 H001 |
| 2 | start_chapter | 起始章节，纯数字 |
| 3 | type | 伏笔类型 |
| 4 | status | 状态（open/暂缓/pressured/near_payoff/resolved/deferred） |
| 5 | last_advanced_chapter | 最近推进章节，建书时填 0 |
| 6 | expected_payoff | 预期回收描述 |
| 7 | payoff_timing | 回收节奏：immediate/near-term/mid-arc/slow-burn/endgame |
| 8 | depends_on | 上游依赖 hook_id 列表，格式 [H003, H007]，无填「无」 |
| 9 | pays_off_in_arc | 回收卷，自然语言（如"第2卷中段"） |
| 10 | core_hook | 是否主线承重：true/false（一本书 3-7 条 core） |
| 11 | half_life | 半衰期（整数章数），可选，不填则按 payoff_timing 推导 |
| 12 | notes | 备注，初始线索放这里不放第5列 |

### 3.4 解析逻辑：parseSectionsWithRepair

解析分两步：先 `parseSections`，失败则 `repairMissingSections` 后重试一次。

#### 步骤 1：parseSections

```
1. parseArchitectSectionMap(content)
   - 先用正则匹配 === SECTION: xxx === 标记
     正则：/^\s{0,3}(?:#{1,6}\s*)?===\s*SECTION\s*[：:]\s*([^\n=]+?)\s*===\s*(?:#+\s*)?$/gim
   - 如果匹配到标记，按标记位置切片成 Map<sectionName, content>
   - 如果没匹配到 === SECTION === 标记，回退到 Markdown 标题匹配
     （canonicalSectionNameFromHeading 把中文标题映射成规范名）
   
2. 规范化 section 名（normalizeSectionName）
   - NFKC 归一化 → 转小写 → 去掉 `"'* _` → 非字母数字转 _ → 去首尾 _

3. 检查必需 section（5 段合约）
   必需：story_frame（或 legacy story_bible）/ volume_map（或 legacy volume_outline）/ roles / book_rules / pending_hooks
   - 如果用了 legacy 名（story_bible/volume_outline）且无新名，允许 roles 为空（v12 兼容）
   - 缺失任一 → 抛 MissingArchitectSectionsError(missing, content)

4. 解析 roles（parseRoles）
   - 按 /^---ROLE---$/m 分块
   - 每块按 /^---CONTENT---$/m 分割 header 和 content
   - header 里提取 tier（major/minor/主要/次要）和 name
   - 畸形条目静默丢弃

5. 规范化 pending_hooks（normalizePendingHooksSection）
   - 解析 Markdown 表格行
   - 解析 13 列（Phase 7 扩展列）
   - 解析 depends_on（[H003, H007] 格式）
   - 解析 core_hook（true/yes/是/核心/core/1/✓）
   - 解析 half_life（可选整数）
   - 根据 shouldPromoteHook 预升级种子伏笔（core_hook / depends_on / cross_volume 三规则）
   - renderHookSnapshot 重新渲染
```

#### 步骤 2：repairMissingSections（修复重试）

如果 `parseSections` 抛出 `MissingArchitectSectionsError`：

```
1. 构建修复 prompt：
   system: "你负责修复 InkOS architect 的输出格式。上一轮草稿有可用内容，但缺少必需的 SECTION 块。
            不要重新发明一本书；保留已有可用内容，只补齐缺失部分并整理成完整输出。
            必须按顺序返回完整 5 段 SECTION：story_frame、volume_map、roles、book_rules、pending_hooks。
            book_rules 必须是普通 Markdown，不要 YAML；pending_hooks 必须是 Markdown 表格。
            不要解释修复过程。"
   user: "缺失 section：{missingList}\n\n原始不完整输出如下：\n\n{content}"

2. LLM 调用（temperature=0.2）

3. 再次 parseSections
   - 成功 → 返回
   - 仍失败 → 抛 ArchitectIncompleteFoundationError（含用户友好提示：
     "基础设定没有生成完整(缺少:xxx)。这通常是模型一次没把所有部分写全，不是你的输入有问题。
      点重试，或换更强的模型(如 deepseek-v4-pro / gpt-5.5)再生成一次。"）
```

### 3.5 落盘文件结构

`writeFoundationFiles` 方法将 `ArchitectOutput` 落盘到书籍目录：

```
<bookDir>/
└── story/
    ├── outline/
    │   ├── story_frame.md          ← 4 段散文（权威来源）
    │   ├── volume_map.md           ← 5 段主体 + 节奏原则尾段（权威来源）
    │   └── 节奏原则.md             ← 仅当 LLM 产出独立块时才写（通常不写）
    ├── roles/
    │   ├── 主要角色/
    │   │   ├── 林辞.md             ← 一人一卡
    │   │   └── ...
    │   └── 次要角色/
    │       ├── 张三.md
    │       └── ...
    ├── story_bible.md              ← 兼容指针（指向 outline/story_frame.md）
    ├── character_matrix.md         ← 兼容指针（指向 roles/ 目录）
    ├── book_rules.md               ← 权威 Markdown 规则卡
    ├── current_state.md            ← 种子占位（运行时 consolidator 每章追加）
    ├── pending_hooks.md            ← 架构师初始伏笔池
    └── emotional_arcs.md           ← 运行时状态（空表头）
```

**落盘规则**：
- `mode=init`：全部文件首次创建，包括 current_state.md 种子占位、pending_hooks.md、emotional_arcs.md 空表头
- `mode=revise`：先删除 roles/ 主要角色和次要角色目录再重建；要求 output 必须是 Phase 5 格式（storyFrame 非空），否则抛错不改文件
- `story_bible.md` 和 `character_matrix.md` 是兼容指针文件，不承载权威内容
- `volume_outline.md` 镜像文件已移除，所有读取方通过 `readVolumeMap()` 解析（优先 outline/volume_map.md，回退 legacy volume_outline.md）

### 3.6 Go 接口与结构体定义

```go
package agents

import "context"

// ArchitectRole 表示一个角色的架构卡。
type ArchitectRole struct {
    Tier    string // "major" 或 "minor"
    Name    string // 角色名
    Content string // 散文角色卡全文
}

// ArchitectOutput 是 ArchitectAgent.generateFoundation 的返回值。
// 同时承载 Phase 5 新字段（StoryFrame/VolumeMap/Roles）和 legacy 兼容字段。
type ArchitectOutput struct {
    // Legacy 字段（兼容外部读取方）
    StoryBible      string // 兼容指针内容，从 StoryFrame 合成
    VolumeOutline   string // 等于 VolumeMap
    BookRules       string // 权威 Markdown 规则卡
    CurrentState    string // 建书时通常为空（consolidator 运行时追加）
    PendingHooks    string // 规范化后的伏笔表快照

    // Phase 5 新字段（权威来源）
    StoryFrame       string           // story_frame section 原文
    VolumeMap        string           // volume_map section 原文
    RhythmPrinciples string           // 通常为空（已合并进 VolumeMap 尾段）
    Roles            []ArchitectRole  // 解析后的角色卡列表
}

// ArchitectAgent 接口定义。
type ArchitectAgent interface {
    // Name 返回 agent 标识符，固定为 "architect"。
    Name() string

    // GenerateFoundation 调用 LLM 生成 5 段 SECTION 基础设定，解析后返回。
    //
    // 参数：
    //   - book: 书籍配置
    //   - externalContext: 外部创作指令（可为空）
    //   - reviewFeedback: 上一轮审查反馈（可为空，非空时触发修复段）
    //   - options: 可选参数，含 ReviseFrom（既有架构稿修订模式）
    GenerateFoundation(
        ctx context.Context,
        book *BookConfig,
        externalContext string,
        reviewFeedback string,
        options *ArchitectOptions,
    ) (*ArchitectOutput, error)

    // GenerateFoundationFromImport 从已有章节反向推导基础设定（续写/系列模式）。
    GenerateFoundationFromImport(
        ctx context.Context,
        book *BookConfig,
        chaptersText string,
        externalContext string,
        reviewFeedback string,
        importMode string, // "continuation" | "series"
    ) (*ArchitectOutput, error)

    // GenerateFanficFoundation 基于原作正典生成同人文基础设定。
    GenerateFanficFoundation(
        ctx context.Context,
        book *BookConfig,
        fanficCanon string,
        fanficMode string, // "canon" | "au" | "ooc" | "cp"
        reviewFeedback string,
    ) (*ArchitectOutput, error)

    // WriteFoundationFiles 将 ArchitectOutput 落盘到 bookDir。
    //
    // 参数：
    //   - bookDir: 书籍根目录
    //   - output: 架构产出
    //   - numericalSystem: 是否有数值系统（仅 API 兼容，实际不影响落盘）
    //   - language: "zh" | "en"
    //   - mode: "init" | "revise"
    WriteFoundationFiles(
        ctx context.Context,
        bookDir string,
        output *ArchitectOutput,
        numericalSystem bool,
        language string,
        mode string,
    ) error
}

// ArchitectOptions 是 GenerateFoundation 的可选参数。
type ArchitectOptions struct {
    ReviseFrom *ReviseFrom // 既有架构稿修订模式
}

// ReviseFrom 用于把已有书的架构稿升级为 Phase 5 段落式。
type ReviseFrom struct {
    StoryBible      string
    VolumeOutline   string
    BookRules       string
    CharacterMatrix string
    UserFeedback    string
}

// ArchitectIncompleteFoundationError 表示 LLM 输出始终不完整（修复重试后仍失败）。
type ArchitectIncompleteFoundationError struct {
    Missing        []string // 缺失的 section 名
    PartialContent string   // 最后一轮的不完整原文
    Message        string   // 用户友好提示
}

func (e *ArchitectIncompleteFoundationError) Error() string {
    return e.Message
}

// missingArchitectSectionsError 内部错误，表示当前轮次解析缺 section。
// 不对外暴露，由 parseSectionsWithRepair 内部捕获并触发修复。
type missingArchitectSectionsError struct {
    Missing []string
    Content string
}
```

**解析相关内部方法签名（Go 风格）**：

```go
// parseSectionsWithRepair 先尝试解析，失败则调 LLM 修复后重试一次。
func (a *architectAgent) parseSectionsWithRepair(ctx context.Context, content, language string) (*ArchitectOutput, error)

// parseSections 解析 5 段 SECTION，缺段抛 missingArchitectSectionsError。
func (a *architectAgent) parseSections(content, language string) (*ArchitectOutput, error)

// parseArchitectSectionMap 把原文切成 map[sectionName]sectionContent。
// 先匹配 === SECTION: xxx ===，无命中则回退到 Markdown 标题匹配。
func (a *architectAgent) parseArchitectSectionMap(content string) map[string]string

// parseRoles 解析 ---ROLE---/---CONTENT--- 块为角色列表。
func (a *architectAgent) parseRoles(raw string) []ArchitectRole

// normalizePendingHooksSection 规范化伏笔表（13 列解析 + 预升级 + 快照渲染）。
func (a *architectAgent) normalizePendingHooksSection(section, volumeMapRaw string) string

// repairMissingSections 调 LLM 补齐缺失 section（temperature=0.2）。
func (a *architectAgent) repairMissingSections(ctx context.Context, missing []string, content, language string) (string, error)
```

## 四、FoundationReviewerAgent 设计

### 4.1 多维度评分

| 规则 | 值 |
|------|-----|
| 总分 | 0-100，5 个维度等权平均（四舍五入） |
| **通过线** | `totalScore >= 80` |
| **单维度下限** | 所有维度 `score >= 60`（任一低于 60 则不通过，即使总分够 80） |
| 通过判定 | `passed = (totalScore >= 80) && !anyBelowFloor` |

5 个维度（original 模式）：

| # | 维度 | 评分问题 |
|---|------|---------|
| 1 | 核心冲突 | 是否有清晰且有足够张力的核心冲突支撑用户要求的 N 章？ |
| 2 | 开篇节奏 | 前 min(5, targetChapters) 章能否形成翻页驱动力？ |
| 3 | 世界一致性 | 世界观是否内洽且具体？ |
| 4 | 角色区分度 | 主要角色的声音和动机是否各不相同？ |
| 5 | 节奏可行性 | 大纲是否适配 N 章，并避免连续 min(10, max(3, target)) 章同一种节拍？ |

### 4.2 三种审查模式

#### 模式 1：original（原创）

使用上述 5 个维度。`targetChapters` 影响维度 1/2/5 的措辞（动态注入章数）。

#### 模式 2：fanfic（同人）

替换为 5 个同人专属维度：

| # | 维度 | 评分问题 |
|---|------|---------|
| 1 | 原作 DNA 保留 | 同人是否尊重原作的世界规则、角色性格、已确立事实？ |
| 2 | 新叙事空间 | 是否有明确的分岔点或新领域，让故事有原创空间，而非复述原作？ |
| 3 | 核心冲突 | 新故事的核心冲突是否有足够张力且区别于原作？ |
| 4 | 开篇节奏 | 前 5 章能否形成翻页驱动力，不需要 3 章铺垫？ |
| 5 | 节奏可行性 | 卷纲是否避免了重走原作剧情节拍的陷阱？ |

#### 模式 3：series（系列续写）

与 fanfic 维度相同，但 `modeLabel` 显示为"系列"。续写方向要求：
- 必须引入新叙事空间（新冲突、新地点、新时间）
- 5 章内引爆
- 50% 以上场景新鲜

### 4.3 审查反馈格式

LLM 必须按以下格式输出：

```
=== DIMENSION: 1 ===
分数：{0-100}
意见：{具体反馈}

=== DIMENSION: 2 ===
分数：{0-100}
意见：{具体反馈}

...（每个维度一个 block）

=== OVERALL ===
总分：{加权平均}
通过：{是/否}
总评：{1-2段总结，指出最大的问题和最值得保留的优点}
```

解析逻辑（`parseReviewResult`）：
- 对每个维度 `i`，用正则匹配 `=== DIMENSION: {i+1} ===` 块内的 `分数/Score: (\d+)` 和 `意见/Feedback: ...`
- 匹配失败时该维度默认 50 分、反馈 "(parse failed)"
- 总分 = 所有维度分数平均（四舍五入）
- `passed = totalScore >= 80 && !any(score < 60)`
- 总评从 `=== OVERALL ===` 块的 `总评/Summary: ...` 提取

**审查输入摘录**（`buildFoundationExcerpt`）：把 ArchitectOutput 的 5 个字段拼成审查 prompt 的 user 消息：
```
## 世界设定
{storyBible}

## 卷纲
{volumeOutline}

## 规则
{bookRules}

## 初始状态
{currentState}

## 初始伏笔
{pendingHooks}
```

fanfic/series 模式额外注入 `## 原作正典参照` 和 `## 原作风格参照` 块。

## 五、generateAndReviewFoundation 审查循环

审查循环最多重试 **2 次**（即最多调用 ArchitectAgent 3 次：1 次初始 + 2 次修复）。

```
maxRetries = 2
attempt = 0
var bestOutput *ArchitectOutput
var bestReview *FoundationReviewResult
var reviewFeedback string

loop:
    attempt += 1

    // 1. 调用 ArchitectAgent 生成基础设定
    output, err := architect.GenerateFoundation(book, externalContext, reviewFeedback, nil)
    if err != nil:
        if attempt <= maxRetries:
            reviewFeedback = "架构师生成失败：" + err.Error()
            goto loop
        return nil, nil, err

    // 2. 调用 FoundationReviewerAgent 审查
    review, err := reviewer.Review(output, mode, fanficCanon, styleGuide, language, book.TargetChapters)
    if err != nil:
        // 审查失败不阻断，用 output 直接落盘（记录警告）
        log.Warn("foundation review failed: " + err)
        return output, nil, nil

    bestOutput = output
    bestReview = review

    // 3. 判定通过
    if review.Passed:
        break  // 通过，落盘

    // 4. 未通过，准备下一轮反馈
    if attempt <= maxRetries:
        reviewFeedback = formatReviewFeedback(review)
        goto loop

// 落盘（无论是否通过，用最优结果落盘）
err = architect.WriteFoundationFiles(bookDir, bestOutput, numericalSystem, language, "init")
return bestOutput, bestReview, err
```

**formatReviewFeedback** 把审查结果格式化为下一轮 ArchitectAgent 的 `reviewFeedback`：

```
维度 1（核心冲突）：75 分
意见：核心冲突张力不足，主角动机过于模糊...

维度 2（开篇节奏）：62 分
意见：前3章铺垫过多...

...

总分：71（未通过）
总评：最大问题是核心冲突不够具体...
```

**重要约束**：
- 审查反馈会注入 ArchitectAgent prompt 的「上一轮审核反馈」段，要求 LLM **明确修复**而非换措辞重写
- 如果 3 次都不通过，仍然落盘最优结果（bestOutput），但返回的 `bestReview.Passed = false`，调用方可决定是否提示用户手动修改
- `ArchitectIncompleteFoundationError`（LLM 始终输出不完整）不触发审查循环，直接返回错误

## 七、LLM prompt 要点（中文版核心段落）

以下是 ArchitectAgent 中文 system prompt 的核心结构（完整 prompt 见 `architect.ts:202-405`）：

### 7.1 开场定位

```
你是这本书的总架构师。你的唯一输出是**散文密度的基础设定**——不是表格、不是 schema、不是条目化 bullet。
v6 以后这本书的"灵气"从哪里来？从你这里来。你的散文密度决定了后面 planner 能不能读出"稀疏 memo"，
writer 能不能写出活人，reviewer 能不能校准硬伤。
```

### 7.2 外部指令段（可选）

```
## 外部指令
以下是来自外部系统的创作指令，请将其融入设定中：

{externalContext}
```

### 7.3 审核反馈段（可选，重试时注入）

```
## 上一轮审核反馈
上一轮基础设定未通过审核。你必须在这次重生中明确修复以下问题，不能只换措辞重写同一套方案：

{reviewFeedback}
```

### 7.4 去重铁律

```
## 去重铁律（必读）
禁止在多段里重复同一事实。主角弧线只写在 roles；世界铁律只写在 story_frame.世界观底色；
节奏原则只写在 volume_map 最后一段；角色当前现状只写在 roles.当前现状；
初始钩子只写在 pending_hooks（startChapter=0 行）。
**如果本书是年代文/历史同人/都市重生等需要年份、季节、重大历史事件作为锚点的题材**，
把环境/时代锚自然织进 story_frame.世界观底色（"1985 年 7 月，非典刚过"这类）；
**修仙/玄幻/系统等没有真实年份的题材直接省略**，不要硬凑。
如果一个段落写了另一段的内容，删掉。
```

### 7.5 预算

```
## 预算（超预算必删）
- story_frame ≤ 3000 chars
- volume_map ≤ 5000 chars
- roles 总 ≤ 8000 chars
- book_rules ≤ 1000 chars（普通 Markdown 规则卡）
- pending_hooks ≤ 2000 chars
```

### 7.6 story_frame 段落要求（节选）

```
=== SECTION: story_frame ===

这是散文骨架。**4 段**，每段约 600-900 字，不要写表格，不要写 bullet list，写成能被人读下去的段落。
**主角弧线不写在本 section；它的权威来源是 roles/主要角色/<主角>.md。**
本段只需一句指针："本书主角是 X，完整弧线详见 roles/主要角色/X.md"。

### 段 1：主题与基调
写这本书到底讲的是什么——不是"讲主角如何从弱到强"这种空话，而是具体的命题。
主题下面跟着基调——温情冷冽悲壮肃杀，哪一种？为什么是这种而不是另一种？

### 段 2：核心冲突、对手定性、前台/后台双层故事
...
**本段必须显式写出"前台故事 / 后台故事"两条线**：
- 前台故事：读者每章看得到的表层冲突
- 后台故事：贯穿全书的暗线——藏在所有前台事件背后的那台"机器"
两条线必须有因果关联，不能是平行宇宙。

### 段 3：世界观底色（铁律 + 质感 + 本书专属规则）
3-5 条不可违反的铁律——以 prose 写出，不要 bullet。感官质感锚。

### 段 4：终局方向 + 全书 Objective
**本段末尾必须明确写出全书 Objective 一句话**：可验证的终局状态。
不要写"变强"、"复仇"这类抽象词，要写一个能被外部观察者判定"达成 / 未达成"的具体状态。
```

### 7.7 volume_map 段落要求（节选）

```
=== SECTION: volume_map ===

只写到卷级 prose——禁止指定具体章号。章级规划是 Phase 3 planner 的职责。

### 段 3：各卷 OKR（Objective + Key Results）
用 OKR 递归大纲法分解全书 Objective。每一卷必须明确给出：
- Objective（卷级目标）：可验证状态，一句话
- Key Results（3 条，可量化）：外部观察者能判定是否完成
**每一卷 3 个 KR 是下游 planner 分解章节任务的直接依据——
planner 拿到一卷的 3 个 KR 后，按每 3-5 章推进一个 KR 的节奏排章。**

### 段 5：节奏原则（具体化 + 通用）
6 条节奏原则。至少 3 条必须具体化到本书。
```

### 7.8 硬性完结检查

```
## 硬性完结检查（生成前读一遍）
必须依次输出全部 5 个 SECTION 块：story_frame → volume_map → roles → book_rules → pending_hooks，
不允许因为 story_frame 或 volume_map 写长了就不写后 3 段。
哪怕 roles 只列 3 个角色、book_rules 只有 Markdown 小块、pending_hooks 只有 3 行，也要完整输出。
只有写完 pending_hooks 最后一行才算交付。
```

### 7.9 修复 prompt（parseSectionsWithRepair 失败时）

```
system:
你负责修复 InkOS architect 的输出格式。
上一轮草稿有可用内容，但缺少必需的 SECTION 块。
不要重新发明一本书；保留已有可用内容，只补齐缺失部分并整理成完整输出。
必须按顺序返回完整 5 段 SECTION：story_frame、volume_map、roles、book_rules、pending_hooks。
book_rules 必须是普通 Markdown，不要 YAML；pending_hooks 必须是 Markdown 表格。
不要解释修复过程。

user:
缺失 section：{missingList}

原始不完整输出如下：
{content}
```

### 7.10 审查 prompt（中文版核心）

```
你是一位资深小说编辑，正在审核一本新书的基础设定（世界观 + 大纲 + 规则）。

你需要从以下维度逐项打分（0-100），并给出具体意见：

1. 核心冲突（是否有清晰且有足够张力的核心冲突支撑用户要求的N章？）
2. 开篇节奏（前5章能否形成翻页驱动力？）
3. 世界一致性（世界观是否内洽且具体？）
4. 角色区分度（主要角色的声音和动机是否各不相同？）
5. 节奏可行性（大纲是否适配N章，并避免连续10章同一种节拍？）

## 评分标准
- 80+ 通过，可以开始写作
- 60-79 有明显问题，需要修改
- <60 方向性错误，需要重新设计

## 输出格式（严格遵守）
=== DIMENSION: 1 ===
分数：{0-100}
意见：{具体反馈}
...
=== OVERALL ===
总分：{加权平均}
通过：{是/否}
总评：{1-2段总结}

审核时要严格。不要因为"还行"就给高分。80分意味着"可以直接开写，不需要改"。
```
